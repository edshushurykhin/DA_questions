## Вопросы по Python

- [ ] Какие типы в Python считаются изменяемыми и неизменяемыми?

К неизменяемым относятся, например, `int`, `float`, `bool`, `str`, `bytes`, `tuple` (если все элементы неизменяемы), `frozenset`. К изменяемым — `list`, `dict`, `set`, большинство пользовательских классов по умолчанию. Неизменяемый объект нельзя «поменять на месте»: операции дают новые объекты, что важно для ключей словарей и для безопасной передачи в многопоточности.

Изменяемые типы удобны для накопления данных в пайплайнах, но их нельзя использовать как ключи `dict` и элементы `set`. На собеседовании полезно связать это с копированием: присваивание не копирует «глубоко» вложенные изменяемые структуры.

- [ ] Чем отличаются операторы `is` и `==`?

`==` сравнивает значения через `__eq__` (семантика равенства). `is` проверяет, указывают ли две переменные на один и тот же объект в памяти (идентичность). Для сравнения с `None` принято писать `x is None`, а не `==`, чтобы не вызывать кастомный `__eq__`.

Для малых целых и интернированных строк интерпретатор может переиспользовать объекты, но на это не стоит опираться в логике. В DE-коде чаще всего `==` для данных, `is` для синглтонов и проверки «тот же ли объект».

- [ ] Назови четыре принципа ООП и зачем они DE на практике.

Инкапсуляция — сокрытие внутреннего состояния за интерфейсом. Наследование — повторное использование и расширение поведения. Полиморфизм — единый интерфейс для разных реализаций. Абстракция — выделение существенного контракта без деталей реализации.

В DE это про проектирование коннекторов, клиентов API и тестируемых DAG-задач: абстракция источника данных позволяет подменять окружение и упрощает сопровождение.

- [ ] Где быстрее искать элемент: в списке или в словаре и почему?

Проверка вхождения или доступ по ключу в `dict` в среднем O(1) за счёт хэш-таблицы. В списке поиск элемента — O(n), потому что нужно просматривать элементы до совпадения (если нет отдельного индекса).

Для больших наборов идентификаторов (например, дедуп по ключу в памяти) словарь или `set` предпочтительнее. Список уместен, когда важен порядок и относительно небольшой объём или нужен последовательный проход.

- [ ] Какие объекты могут быть ключами словаря в Python?

Ключом может быть только хэшируемый (hashable) неизменяемый объект: чаще `str`, `int`, `tuple` из хэшируемых элементов, `frozenset`. Список, словарь и множество ключами быть не могут, потому что они изменяемы.

Если нужен составной ключ, обычно используют кортеж из примитивов или явный хэш/строковое представление. Это напрямую перекликается с проектированием ключей в DWH.

- [ ] Меняется ли порядок ключей в словаре в современных версиях Python?

С Python 3.7 язык гарантирует сохранение порядка вставки в `dict` (раньше это было CPython-деталью, теперь часть спецификации). Перебор `dict` отражает порядок добавления ключей.

Для детерминированного вывода и сравнения конфигов это удобно, но логику приложения лучше не строить на «случайном» порядке без явной сортировки, если нужен фиксированный порядок полей.

- [ ] Может ли кортеж быть ключом словаря?

Да, если кортеж содержит только хэшируемые элементы. Кортеж с вложенным списком ключом быть не может.

На практике кортежи-ключи используют для группировки по нескольким полям в чистом Python до загрузки в БД.

- [ ] Что значит «объект хэшируемый» и зачем это словарю?

Хэшируемый объект имеет стабильный `__hash__` и реализует `__eq__` для сравнения. Словарь кладёт ключ в корзину по хэшу и разрешает коллизии через сравнение на равенство.

Если бы изменяемые объекты были ключами, хэш и содержимое расходились бы, и структура словаря ломалась бы. Поэтому изменяемые типы исключены.

- [ ] Что такое контекстный менеджер и зачем он в DE-коде?

Это объект, используемый в конструкции `with`, который гарантирует выполнение кода при входе и выходе из блока (в том числе при исключениях): открытие/закрытие файла, соединения с БД, транзакции, временные ресурсы.

В пайплайнах это снижает риск «забыть закрыть» соединение или файл при ошибке и делает код читаемее, чем ручной `try/finally`.

- [ ] Как реализовать контекстный менеджер в Python?

Два пути: класс с методами `__enter__` и `__exit__`, либо генератор с декоратором `@contextmanager` из `contextlib`, где код до `yield` — вход, после `yield` — выход и очистка.

`__exit__` получает информацию об исключении; можно подавить его, вернув `True`, но это делают осознанно. Для простых случаев `contextlib.closing` и готовые менеджеры библиотек часто достаточно.

- [ ] Какие методы нужны у класса-контекстного менеджера?

`__enter__(self)` возвращает ресурс (или `self`). `__exit__(self, exc_type, exc, tb)` вызывается при выходе из блока; в нём закрывают ресурс и при необходимости обрабатывают исключение.

Это стандартный протокол CPython; понимание `__exit__` показывает зрелость при работе с транзакциями и файлами.

- [ ] Чем `@classmethod` отличается от `@staticmethod`?

`classmethod` получает первым аргументом класс (`cls`) и может обращаться к состоянию класса и вызывать альтернативные конструкторы. `staticmethod` — обычная функция внутри класса без привязки к `self`/`cls`, просто сгруппированная по namespace.

Фабричные методы создания объекта из конфига или строки часто делают `@classmethod`. `staticmethod` уместен для чистых утилит, логически относящихся к классу.

- [ ] В чём разница между итератором и генератором?

Итератор — объект с `__next__` и протоколом итерации; генератор — функция с `yield`, из которой Python строит итератор автоматически. Генератор лениво отдаёт значения и экономит память.

Для потоковой обработки больших файлов и батчей генераторы предпочтительнее материализации всего списка в память.

- [ ] Что делают `*` и `**` в определении функции и при её вызове?

В определении `*args` собирает лишние позиционные аргументы в кортеж, `**kwargs` — лишние именованные в словарь. При вызове `*sequence` распаковывает итерируемое в позиционные параметры, `**mapping` — в именованные.

Это часто встречается в обёртках над Airflow/Spark и при сборке динамических конфигов; важно не путать с аннотациями вида `list[str]` в новых версиях Python.

- [ ] Почему общий изменяемый атрибут на уровне класса даёт «шаринг» между экземплярами?

Атрибут, объявленный в теле класса (например `lst = []`), создаётся один раз и хранится в объекте класса; если в `__init__` делают `self.lst.append(...)` без переназначения `self.lst`, все экземпляры пишут в один и тот же список.

Для DE это источник тонких багов в stateful-хелперах; правильный паттерн — `self.lst = []` в `__init__` или `dataclasses.field(default_factory=list)`.

## Архитектура и общий стек

- [ ] Как выглядит end-to-end pipeline в современном DE-стеке?

End-to-end — это полный путь данных «от источника до отчёта или модели». Типично его описывают так: источник (чаще операционная OLTP-база, сервисы, файлы или поток событий) → забор данных (ingestion): либо пакетами по расписанию, либо CDC — непрерывный поток изменений из журнала транзакций БД → сырой слой в data lake (файлы в объектном хранилище, например S3-совместимое), где данные сохраняются почти без потерь для аудита → слой качества и унификации (silver, core): единые сущности, дедуп, типизация → витрины (gold, mart) в DWH или OLAP-движке → потребители (BI, API, обучение ML). На интервью важно упомянуть не только цепочку, но и владение качеством: кто мониторит свежесть, кто принимает решение о перезапуске и backfill, кто получает алерты.

Разделение raw / silver / gold объясняют так: raw нужен для воспроизводимости — можно пересчитать верхние слои после смены правил без повторного выдирания истории из источника (reprocessing). silver — зона контрактов и очистки, чтобы несколько команд не считали разные версии правды. gold — стабильные бизнес-метрики с понятным grain (уровнем детализации строки). Такой ответ показывает, что вы понимаете эксплуатацию, а не только «нарисовал стрелочки».

- [ ] Из каких компонентов состоит production Airflow?

Apache Airflow в продакшене — это несколько служб с разными ролями. Metadata database (часто PostgreSQL) хранит все состояние: описания DAG, запуски, статусы задач, переменные, connections. Scheduler периодически смотрит, какие задачи нужно поставить в очередь, и создаёт task instance. Workers (или поды в Kubernetes) выполняют код задач. Webserver отдаёт UI и API для операторов. При deferrable-сенсорах и ожиданиях добавляют triggerer, чтобы долгое ожидание не занимало воркер. В одном предложении: Airflow знает «что и когда запустить» и сохраняет историю — сам тяжёлый расчёт делают Spark, БД или внешние сервисы.

На собеседовании полезно чётко сказать: Airflow — оркестратор (orchestration), а не движок ETL. Он не заменяет Spark или SQL-сервер и не место для хранения больших массивов данных; большие результаты передаются ссылкой (путь в S3, таблица в БД).

- [ ] Из чего состоит Kafka-кластер и что делает каждый компонент?

Apache Kafka можно представить как распределённый журнал событий: данные упорядочены внутри partition, а несколько партиций дают парал­лелизм. Broker — сервер хранения и обслуживания логов. Topic — именованный поток сообщений логически; partition — физический кусок topic с упорядоченным offset (номер записи в партиции). Producer публикует записи с опциональным key для выбора партиции. Consumer читает в рамках consumer group: партиции распределяются между участниками группы так, чтобы каждую партицию внутри группы читал максимум один consumer. Так масштабируют потребление.

В новых версиях кластер часто переводят на KRaft для хранения метаданных кластера вместо ZooKeeper (служба координации прошлых поколений). Для связи Kafka с базами без самописного кода ставят Kafka Connect — фреймворк коннекторов «источник/приёмник»; для CDC из БД часто используют Debezium.

- [ ] Какие роли у driver и executor в Spark?

Apache Spark разделяет координатора и исполнителей. Driver — процесс приложения на JVM, который знает код и конфигурацию; он строит план выполнения (через DAG задач), договаривается с cluster manager (YARN/K8s и т.д.), рассылает tasks исполнителям и может держать broadcast у себя перед раздачей. Executor — процесс на рабочем узле, который реально считает task над partition, кэширует данные если задан cache/persist, пишет и читает shuffle-файлы (промежуточный обмен при «широких» операциях). Если на driver ошибочно тянуть гигантский dataset командой `collect()`, он упадёт по памяти или сети, даже когда executors ещё свободны.

Практический тезис: данные должны обрабатываться рядом с хранением узлами кластера, а не «стягиваться» на одну машину без необходимости.

- [ ] Как устроен кластер Greenplum на высоком уровне?

Greenplum — это MPP-СУБД в духе PostgreSQL, но распределённая: есть master (координатор), который принимает SQL, строит распределённый план и оркестрирует выполнение на segment-хостах. Сегменты хранят партиции таблиц по выбранному distribution key (ключу распределения). Между сегментами данные пересылаются через motion (сетевой обмен) — это часто самое дорогое в запросе. Производительность сильно зависит от того, как равномерно строки разошлись по сегментам и насколько join можно выполнить локально без лишнего перераспределения.

Важный месседж: Greenplum — не «немного увеличенный одиночный Postgres», а платформа с отдельными правилами моделирования таблиц, статистикой и планированием запросов.

- [ ] Как устроен ClickHouse-кластер?

Типично данные живут в локальных таблицах движка MergeTree на каждом shard (шарде — доля данных по ключу). Реплики одного шарда дают отказоустойчивость и разгрузку чтения; их согласование часто через Keeper/ZooKeeper. Поверх для запросов создают `Distributed`-таблицу — это прокси: при `SELECT` она отправляет работу на шарды и собирает результат. При вставках тоже важно понимать, куда именно улетают строки (локальная таблица vs распределённый маршрут).

На интервью стоит назвать то, без чего модель нежизнеспособна: `ORDER BY` задаёт порядок и sparse-отсечение гранулами, `PARTITION BY` — логические куски по времени или домену, шардирование должно совпадать с типичными фильтрами. Иначе «ClickHouse сам быстрый» не поможет — будут сканы через весь объём или лишний сетевой обмен между шардовыми узлами.

## SQL

- [ ] Чем **WHERE** отличается от **HAVING**?

**`WHERE`** отбирает отдельные строки ещё до того, как сработает `GROUP BY` и агрегатные функции вроде `SUM`, `COUNT`. 

**`HAVING`** фильтрует уже готовые группы — то есть применяется после агрегации и может ссылаться на результат агрегатов (`HAVING COUNT(*) > 1`). 
Поэтому условия на «сырые» колонки почти всегда кладут в `WHERE`: так СУБД отсекает лишнее раньше и дешевле.

*Типичная ловушка на собеседовании:* написать `WHERE SUM(amount) > 0`: агрегат в `WHERE` в стандартном SQL недоступен именно из-за порядка логических шагов. Если вы проговариваете упрощённую последовательность «сначала строки, потом группы», это звучит уверенно.

- [ ] Какой логический порядок выполнения операторов в **SELECT**?

Упрощённо (как учат на курсах) порядок такой: **`FROM` → `JOIN` → `WHERE` → `GROUP BY` → `HAVING` → оконные функции с `OVER(...)` → список `SELECT` (проекция и выражения) → `ORDER BY` → `LIMIT`/`OFFSET`**. 

Из этого следует, например, что в `WHERE` нельзя ссылаться на алиасы из `SELECT` (во многих СУБД), зато `ORDER BY` часто может — потому что выполняется позже.

*Важная оговорка для интервью:* это логический порядок для понимания семантики. Физический план (query plan), который строит оптимизатор, может переупорядочить операции (например, протолкнуть фильтры ближе к скану), если результат остаётся тем же самым.

- [ ] Какие JOIN нужно знать и как их объяснять на интервью?

Минимальный набор *логических* `JOIN` в SQL: 

**SELF JOIN** - Таблица соединяется сама с собой. **min=0, max=|A|² (зависит от условия ON)**

**INNER JOIN** - Только строки, где есть совпадение в обеих таблицах. **min=0, max=min(|A|,|B|)**

**LEFT JOIN** - Все строки из левой + совпадения из правой. unmatched → NULL. **min=|A|, max=|A|**

**RIGHT JOIN** - Все строки из правой + совпадения из левой. unmatched → NULL. **min=|B|, max=|B|**

**FULL JOIN** - Все строки из обеих таблиц. unmatched → NULL с нужной стороны. **min=max(|A|,|B|), max=|A|+|B|**

**CROSS JOIN** — декартово произведение всех комбинаций (без `ON`), его используют редко и осознанно из‑за взрыва объёма. **min=0, max=|A|×|B|**

*Отдельно проговаривают кардинальность (cardinality):* это «сколько строк справа в среднем приходится на одну строку слева». Если связь не 1:1, при `JOIN` число строк может вырасти (эффект fan-out), и суммы в отчёте задвоятся. Поэтому перед **JOIN** проверяют grain (что одна строка результата означает) и делают промежуточную агрегацию «многих» к нужному ключу.

- [ ] Что такое физические JOIN (Hash/Merge/Nested Loop) и когда каждый используется?

*Логический* `JOIN` в SQL — это то, что вы хотите получить. 
*Физический* `JOIN` — это как движок это сделает. 

Три классических варианта: 

**Hash Join** — строит хэш-таблицу по одной стороне (обычно по меньшей) и «пробивает» вторую; хорош на больших объёмах при равенстве ключей. 

  *Когда:* Большие таблицы без индексов/сортировки, только equi-join (=).

  *Сложность:* Время O(N+M), Память O(min(N,M)).
  
**Merge Join** — обе стороны отсортированы по ключу (или сортируются); эффективен, если порядок уже есть или сортировка дешёвая (дешевле хеширования). 
 
  *Когда:* Данные уже отсортированы по ключу или сортировка дешевле хеширования; поддерживает range-join и неравенства.

  *Сложность:* Время O(N log N + M log M) (с учётом сортировки), Память O(N+M) (буферы для сортировки).
  
**Nested Loop** — для каждой строки внешней таблицы ищет совпадения во внутренней (цикл в цикле). 
  
  *Когда:* Уместен, когда внешняя сторона маленькая или есть индекс для быстрого поиска во внутренней.

  *Сложность:* Время O(N×M), Память O(1).

Универсального «лучшего `JOIN`» нет: всё зависит от размеров, селективности фильтров, индексов и статистики. Поэтому смотрят фактический план, а не только текст SQL.

- [ ] Что такое оконные функции и когда они нужны?

**Оконная функция (Window function)** считает значение по набору строк, заданному фразой `OVER (...)`, но не схлопывает весь результат до одной строки на группу, как `GROUP BY`. 

В `OVER` обычно указывают `PARTITION BY` (границы «окна» по группам, например по `user_id`) и `ORDER BY` (порядок внутри окна). 

Оконные функции разделяются на:

  - **Агрегатные** (SUM, AVG, MIN, MAX)
  - **Ранжирующие** (RANK, DENSE_RANK, ROW_NUMBER)
  - **Функции смещения** (LAG, LEAD, FIRST_VALUE, LAST_VALUE)

*В витринах это стандарт:* взять последнюю версию записи на сущность без `GROUP BY`, который отбрасывает детали, или посчитать кумулятивную сумму без тяжёлого self-join.

- [ ] Как корректно дедуплицировать данные SQL-ом?

*Частый приём:* пронумеровать дубликаты внутри бизнес-ключа и оставить одну строку.

Пример идеи: `ROW_NUMBER() OVER (PARTITION BY business_key ORDER BY updated_at DESC, id DESC) AS rn`, затем внешний запрос `WHERE rn = 1`. 

Второй и третий критерии в `ORDER BY` нужны для детерминизма: если у двух строк совпал `updated_at`, вы всё равно выбираете однозначного «победителя».

*На собеседовании важно сказать бизнес-правило:* что считается «актуальной» версией (последний штамп времени, максимальный `version`, строка не помечена `deleted` и т.д.). Без этого SQL может быть формально верным, а цифры — чужие бизнесу.

- [ ] **UNION** и **UNION ALL** — что выбрать?

**UNION** по стандарту убирает дубликаты между двумя наборами — для этого СУБД обычно делает дополнительную работу (hash или sort уникальности), что дороже по CPU и памяти. 

**UNION ALL** просто склеивает результаты без удаления повторов и почти всегда быстрее.

В data pipeline чаще пишут `UNION ALL`, а осознанный dedup выносят отдельным шагом с явными правилами — так проще отлаживать и сравнивать row count до/после.

- [ ] Операции со множествами в SQL

Операции со множествами позволяют **комбинировать** (именно комбинировать, не объединять, как при JOIN) результаты нескольких запросов друг с другом, и получить один общий результат.

**UNION** - объединяет записи из двух запросов в один общий результат.

**EXCEPT** - возвращает все записи, которые есть в первом запросе, но отсутствую во втором (разница множеств).

**INTERSECT** - возвращает все записи, которые есть и в первом запросе, и во втором (пересечение множеств).

- [ ] Как читать **EXPLAIN ANALYZE**?

**EXPLAIN** показывает план запроса — какие операции собирается выполнить оптимизатор. 

Вариант **EXPLAIN ANALYZE** (в PostgreSQL и аналогах) дополнительно выполняет запрос и пишет фактическое время и число строк по узлам. 

Смотрите сверху вниз или по самым дорогим узлам: 

  - Seq Scan (полный перебор), Sort, Hash (построение хэш-таблицы), Nested Loop, а в MPP (Greenplum и др.) — ещё motion/пересылки между узлами.
  
  - Сравните оценённое число строк (estimate) с реальным (actual): большой разрыв часто значит устаревшую статистику. *После крупных загрузок помогает `ANALYZE`*. 
  
*Сильный ответ*: «я сначала ищу узкое горлышко по времени и строкам, потом проверяю, не ошибся ли план из‑за статистики или неудачного join».

- [ ] Когда **NOT EXISTS** лучше **NOT IN**?

Для anti-join («взять строки левой таблицы, которых нет в правой») часто предпочитают коррелированный подзапрос с `NOT EXISTS`. Проблема `NOT IN (подзапрос)`: если подзапрос вернул хотя бы один `NULL` в списке значений, семантика трёхзначной логики SQL даст UNKNOWN для сравнения с `NULL`, и в итоге `NOT IN` может не вернуть ни одной строки — поведение кажется «магическим». `NOT EXISTS` не страдает от этой ловушки в том же виде и обычно предсказуемее.

- [ ] Почему **WHERE col NOT BETWEEN a AND b** может не отобрать ни одной строки, хотя «очевидно» есть выбросы?

В SQL сравнения с `NULL` не дают ни `TRUE`, ни `FALSE`, а дают третье значение — `UNKNOWN`. Строка попадает в результат `WHERE`, только если условие стало `TRUE`. Если `col` — `NULL`, то выражение `BETWEEN a AND b` для этой строки даёт `NULL`, а `NOT (NULL)` снова `NULL`, не `TRUE` — строка отфильтровывается. То же для `col = NULL` (нужно `IS NULL`) и `col > NULL`.

*Практический вывод: явно проектируйте работу с пропусками:* `IS NULL`, `COALESCE`, отдельные правила «что делать с неизвестным значением» в витрине.

- [ ] Как оценить число строк в **`INNER` / `LEFT` / `FULL` / `CROSS`** JOIN на маленьком примере без прогона?

Сначала зафиксируйте уникальность ключа и тип связи (1:1, 1:N, N:M). Учебный пример: в `tab1` 5 различных `id`, в `tab2` 5 различных `id`, пересечение по ключу — 3 значения. Тогда:

  - `INNER JOIN` даст 3 строки (только совпавшие ключи). 
  
  - `LEFT JOIN` сохранит все 5 строк слева (у двух «лишних» ключей справа будут NULL). Аналогично `RIGHT` от зеркальной стороны. 
  
  - `FULL OUTER JOIN` по уникальным ключам даст 5 + 5 − 3 = 7 строк (все ключи из объединения без дублирования одного ключа дважды). 
  
  - `CROSS JOIN` даст 5×5 = 25 комбинаций — это декартово произведение без условия связи.

Если ключ не уникален, число строк растёт как сумма произведений совпадений по каждому значению ключа. Смысл вопроса — вы умеете мыслить grain-ом и кардинальностью, а не только синтаксисом `JOIN`.

- [ ] Как избежать **Double counting** в витринах?

**Double counting (двойной счёт)** чаще всего появляется, когда к факту с grain «одна строка = одно событие» присоединяют таблицу со связью один-ко-многим без предварительного сжатия «многих» к одной строке на ключ факта. 

*Лекарство:* сначала агрегировать «правую» сторону до того же ключа, что и факт (или до нужного grain), потом `JOIN`. Параллельно документируют тип связи и проверяют число строк после каждого `JOIN`.

*После сборки витрины полезны контрольные суммы:* сравнение итоговых SUM/COUNT с отчётом из источника, проверки на аномальные доли и дубликаты ключей.

- [ ] Когда использовать CTE, а когда временную таблицу?

**CTE (`WITH`)** — это именованный подзапрос внутри одного оператора `SELECT`/`INSERT`/… Он улучшает читаемость, помогает строить рекурсию и разбивать логику на шаги без отдельного объекта в каталоге. 

**Временная таблица (`TEMP TABLE`)** — физический объект сессии (или промежуточное хранение), к которому можно обращаться несколько раз, на неё проще вешать индекс и статистику, и иногда планировщик ведёт себя стабильнее на тяжёлых MPP.

На Greenplum/Vertica и др. длинный «монолит» из CTE иногда хуже по spill и motion, чем два шага с temp и `ANALYZE` между ними — но это уже тюнинг под конкретную СУБД.

- [ ] Чем VIEW отличается от CTE (`WITH`)?

**VIEW** - Представление

  - *Суть:* Виртуальная таблица, хранит только SQL-запрос.

  - *Хранение:* Нет. Данные читаются из базовых таблиц при каждом SELECT.

  - *Актуальность:* Всегда real-time.

  - *Кейс:* Упрощение сложных запросов, ограничение доступа к данным, переиспользование логики без дублирования.

**MATERIALIZED VIEW** - Материализованное представление

  - *Суть:* Физический снимок результата запроса, сохранённый на диске.

  - *Хранение:* Да. Занимает место как обычная таблица.

  - *Актуальность:* Устаревает до следующего REFRESH (Postgres) или обновляется инкрементально/асинхронно (ClickHouse).

  - *Кейс:* Ускорение тяжёлых агрегаций, BI-дашборды, отчёты, где допустима задержка актуальности.

**CTE** 

  - *Суть:* Временный именованный набор данных, существующий только в рамках одного запроса.

  - *Хранение:* Нет. Не сохраняется в БД (оптимизатор может подставить код inline или временно держать в памяти).

  - *Актуальность:* Только на время выполнения запроса.

  - *Кейс:* Структурирование сложного SQL, пошаговые вычисления, рекурсивные запросы. Не подходит для кэширования между сессиями.

*Запомнить одной строкой:* VIEW = логика без данных | MATERIALIZED VIEW = данные + логика (требует обновления) | CTE = временная переменная внутри одного запроса.

*Нюанс:* в некоторых СУБД CTE может быть встроен (inlined) или материализован по решению оптимизатора — от этого зависит, помогает ли CTE производительности или мешает.

- [ ] При каких условиях VIEW в SQL остаётся «обновляемым» (updatable)?

**Updatable view** — когда через `INSERT`/`UPDATE`/`DELETE` по view СУБД однозначно понимает, какую строку базовой таблицы менять. Обычно это одна базовая таблица без `GROUP BY`, `DISTINCT`, произвольных выражений, которые нельзя обратить, и без «опасных» `JOIN` (правила зависят от диалекта). Нужна понятная связь строки view со строкой таблицы (часто через ключ/уникальный набор колонок).

В DE витрины чаще пишут процессом загрузки в таблицу‑факт, но знание updatable view полезно для staging и для понимания security представлений.

- [ ] В чем разница **DELETE** и **TRUNCATE** в практическом DE-контексте?

**DELETE ... WHERE** удаляет строки по условию, пишет журнал/MVCC-нагрузку по строкам и может включать триггеры — на больших объёмах часто медленно. 

**TRUNCATE** для целой таблицы обычно дешевле: отбрасывает все страницы/экстенты разом (детали зависят от СУБД), но это почти всегда операция «всю таблицу», не точечная. 

В аналитике вместо ночного `DELETE` по всей истории часто делают перезапись партиции (partition swap / overwrite partition) — проще воспроизвести загрузку и контролировать SLA.

- [ ] Чем оконные функции отличаются от обычных агрегатов с **GROUP BY**?

`GROUP BY` схлопывает входные строки: на выходе меньше строк, по одной (или нескольким при grouping sets) на комбинацию группирующих колонок. Оконная функция считает агрегат по набору соседних строк (frame), но каждая исходная строка остаётся в результате — меняется только добавленным столбцом. `PARTITION BY` задаёт «границы окна», `ORDER BY` — порядок обработки.

*Итого:* `GROUP BY` — когда нужен итог на группу; окна — когда нужны ранги, скользящие суммы, `LAG`/`LEAD`, дедуп с сохранением исходных полей.

- [ ] Чем отличаются `COUNT(*)` и `COUNT(колонка)`?

**COUNT(*)** в группе считает все строки, независимо от `NULL` в колонках. 

**COUNT(column)** считает строки, где `column IS NOT NULL` — то есть отвечает на вопрос «сколько непустых значений», а не «сколько всего строк». Это различие критично в DQ (data quality): например, «число заказов» vs «число заказов с известным `customer_id`».

- [ ] Когда hash-join быстрее nested loop, а когда наоборот?

**Hash Join быстрее Nested Loop, если:**

  - *Нет индексов:* Таблицы большие и не отсортированы.

  - *Нужно обработать всё:* Требуется вернуть все или большинство строк (Full Scan).

  - *Условие равенства:* Соединение по = (equi-join).

  - *Почему:* Сложность O(N + M) (линейная), против O(N × M) (квадратичная) у Nested Loop без индексов.

**Nested Loop быстрее Hash Join, если:**

  - *Есть индекс:* Во внутренней таблице есть индекс по ключу соединения.

  - *Нужно мало строк (LIMIT):* Hash Join обязан сначала построить всю хеш-таблицу (читает всё), а Nested Loop может начать отдавать строки сразу и остановиться после нахождения нужного количества.

  - *Почему:* Поиск по индексу работает за O(log N) или O(1), что делает цикл быстрым при малом количестве итераций.

**Итог:**

Hash Join = «Тяжёлая артиллерия» для больших данных без индексов.

Nested Loop = «Точечный удар» для малых выборок или при наличии хороших индексов.

Реальный выбор делает оптимизатор на основе статистики; интервьюеру важно услышать логику компромисса, а не названия.

- [ ] Можно ли заставить оптимизатор выбрать другой JOIN (хинты) и стоит ли?

**Хинты (hints)** или сессионные флаги вроде PostgreSQL `enable_hashjoin=off` — способ принудить или запретить вариант плана для диагностики и редких случаев, когда план стабильно плохой. В боевом коде вечные хинты без измерений опасны: данные меняются, статистика обновится — «спасательный» хинт станет тормозом.

Правильный порядок: `EXPLAIN (ANALYZE, ...)`, `ANALYZE` таблиц, переписать запрос, партиции/индексы — и только потом аккуратно рассмотреть хинты.

- [ ] Чем **DROP TABLE** отличается от **TRUNCATE** и **DELETE**?

**DROP TABLE** удаляет объект таблицы из каталога (metadata) вместе с данными (формулировка зависит от СУБД); после этого обращаться к таблице нельзя, пока не создадите заново. 

**TRUNCATE** очищает строки, сама таблица и GRANT/часть зависимостей остаются (ограничения и `FK` могут блокировать операцию — смотреть документацию). 

**DELETE** — строчное удаление с `WHERE` и полной дорогой MVCC/логирования там, где оно нужно.

- [ ] Таблица A — 10 строк, таблица B — 100 строк. Какие минимальное и максимальное число строк даёт **FULL OUTER JOIN** по ключу?

Ответ без уточнений предположений некорректен — это и проверяется на интервью. Если ключи в обеих таблицах уникальны и не пересекаются, получится 10 + 100 = 110 строк (`FULL OUTER`: все ключи из объединения, для отсутствующей половины NULL). Если связь один-к-одному для 10 строк A, которые совпадают с частью B, возможны тонкие сценарии; крайность «один общий ключ у всех строк» даст декартово 10×100 = 1000 строк (кардинальный fan-out одного значения).

*Главное — проговорить уникальность ключа и тип связи до того, как называть число.*

- [ ] Какие **типы индексов** в PostgreSQL стоит перечислить на интервью?

**B-tree** — стандарт для =, <, >, BETWEEN, LIKE 'prefix%'. Поддерживает сортировку. (PostgreSQL, MySQL, Oracle — по умолчанию).

**Hash** — только точное совпадение (=). Очень быстрый, но не поддерживает диапазоны и сортировку.

**Bitmap** — битовая карта для колонок с низкой кардинальностью (пол, статус). Эффективен для сложных AND/OR условий. (ClickHouse, Oracle).

**GiST / GIN** — для полнотекстового поиска, JSON, массивов, геоданных. GIN быстрее на чтение, GiST — на запись. (PostgreSQL).

**BRIN (Block Range Index)** — хранит мин/макс значения для блоков страниц. Дешёвый и компактный для больших отсортированных таблиц (временные ряды). (PostgreSQL, ClickHouse).

**Columnar / Sparse Index** — индекс по ключевым колонкам для пропуска целых блоков данных. Работает вместе с партиционированием. (ClickHouse PRIMARY KEY).

**Covering / Included Index** — индекс, который включает не только ключевые, но и дополнительные колонки, позволяя удовлетворить запрос без обращения к таблице (Index-Only Scan).

**Partial / Filtered Index** — индекс только для подмножества строк (WHERE status = 'active'). Экономит место и ускоряет запросы с тем же фильтром.

*Как отвечать одной строкой:* «B-tree — универсальный для диапазонов и точных совпадений; Hash — только для =; Bitmap — для низкокардинальных колонок; GIN/GiST — для JSON, текста и геоданных; BRIN — для больших отсортированных таблиц; Covering — для Index-Only Scan; Partial — для фильтрации по условию.»

## Реляционные БД, OLTP/OLAP, транзакционность

- [ ] Что такое **ACID** и почему это важно?

**ACID** — набор свойств транзакции (логически связанного набора операций «всё или ничего») в реляционных СУБД. 

**A — Atomicity (атомарность):** либо выполняются все изменения транзакции, либо откатываются все — нет «половины заказа». 

**C — Consistency (согласованность):** после COMMIT база остаётся в состоянии, удовлетворяющем правилам (constraints, FK). 

**I — Isolation (изоляция):** параллельные транзакции не должны «ломать» друг друга неожиданными промежуточными чтениями (уровень изоляции уточняет, насколько строго). 

**D — Durability (долговечность):** зафиксированные данные переживут сбой процесса при типичных гарантиях журнала.

Для DE это важно при чтении из OLTP: вы понимаете, видите ли вы закоммиченные строки, не попадёте ли в грязное чтение при длинной выгрузке и как изоляция влияет на снимок данных в момент `SELECT`.

- [ ] Что такое уровни изоляции транзакций простыми словами?

**Read Uncommitted**

  - *Суть:* Можно читать «грязные» (незакоммиченные) данные из других транзакций.

  - *Проблемы*: Dirty Read, Non-Repeatable Read, Phantom Read.

  - *Кейс*: Приблизительная аналитика, где допустима неточность (редко используется).

**Read Committed**

  - *Суть*: Видны только закоммиченные данные. Каждый SELECT видит новый снапшот.

  - *Проблемы*: Non-Repeatable Read, Phantom Read.

  - *Кейс*: Большинство OLTP-систем, где важна актуальность данных здесь и сейчас.

**Repeatable Read** 

  - *Суть*: Транзакция видит данные такими, какими они были на момент первого чтения. Гарантирует повторную читаемость.

  - *Проблемы*: Теоретически возможен Phantom Read (но в MySQL решен через Next-Key Locking).

  - *Кейс*: Финансовые операции, где критично, чтобы данные не изменились в процессе логической обработки.

**Serializable**

  - *Суть*: Полная изоляция. Транзакции выполняются последовательно, как будто одна за другой.

  - *Проблемы*: Нет аномалий, но высокий риск блокировок и ошибок сериализации.

  - *Кейс*: Критичные системы с высокой конкуренцией за одни и те же строки (букинг мест, банковский учёт).

*Как отвечать одной строкой:* Изоляция регулирует видимость изменений между транзакциями. Read Uncommitted допускает грязное чтение. Read Committed (стандарт) видит только закоммиченные данные, но допускает неповторяемое чтение. Repeatable Read гарантирует стабильность данных в рамках одной транзакции. Serializable полностью изолирует транзакции ценой производительности. На практике чаще всего использую Read Committed как баланс между консистентностью и конкурентностью 

- [ ] Что такое **нормализация** и **денормализация** данных?

**Нормализация** — разбиение таблиц так, чтобы убрать избыточное дублирование и аномалии обновления (когда одно и то же «фактическое» значение хранится в нескольких местах и расходится). Это типичный дизайн OLTP, где важны целостность и частые точечные правки.

**Денормализация** — намеренный откат: дублируют или «широко» склеивают колонки, чтобы уменьшить число `JOIN` и ускорить чтение отчётов. В DWH денормализация нормальна для витрин; цена — сложнее ETL и выше риск рассинхрона, если нет дисциплины загрузки.

- [ ] Какие первые три нормальные формы нужно знать?

- **1NF**: в ячейке атомарное значение (не «список внутри ячейки» без отдельной таблицы), нет повторяющихся групп столбцов «массивом в одной строке». 

- **2NF**: при составном ключе каждый неключевой атрибут зависит от всего ключа целиком, а не от его части (убираем частичные зависимости). 
  
- **3NF**: неключевые атрибуты зависят только от ключа, а не «друг от друга» транзитивно (убираем скрытые зависимости между описательными полями).

На интервью обычно не ждут формулировок из учебника, а проверяют, какую проблему вы решаете: дубли, противоречивые обновления, лишние `JOIN`.

- [ ] Какие типы индексов и структуры индексов нужно понимать?

Базовая структура в классических СУБД — B-tree (сбалансированное дерево): поиск по равенству и диапазону, помощь `ORDER BY`. 

Hash-структуры полезны для точного равенства, но хуже для диапазонов. 

В аналитических DWH встречаются bitmap-индексы для низкокардинальных фильтров. 

В PostgreSQL дополнительно GIN/GiST/BRIN под специфические типы данных.

*Цена любого индекса — нагрузка на запись (INSERT/UPDATE/DELETE), место на диске, VACUUM/обслуживание. Поэтому индексируют под профиль запросов, а не «все столбцы подряд».*

- [ ] **OLTP** и **OLAP** — в чем ключевое различие?

**OLTP (Online Transaction Processing)** — множество коротких операций: оплата, заказ, просмотр карточки по идентификатору. Часто нормализованная модель, важна низкая задержка и строгая транзакционность при записи.

**OLAP (Online Analytical Processing)** — тяжёлые `SELECT` с `GROUP BY`, `JOIN` больших исторических массивов, полные сканы по партициям. Модели ближе к звезде/витринам, хранилища часто колоночные или MPP.

*Прямые OLAP‑запросы в боевую OLTP обычно запрещают: они конкурируют за CPU/IO и locks и ухудшают продукт.*

- [ ] Как в двух фразах отличить «обычную БД» от хранилища данных (DWH)?

**Операционная БД** обслуживает работу продукта с пользователем: много мелких транзакций, мало JOIN между десятками таблиц в одном запросе от конечного UI. 

**DWH** служит для решений и отчётности: собирают данные из множества источников, хранят историю (SCD, снимки), публикуют витрины под BI, SLA измеряют в «данные за вчера к 09:00», а не в миллисекундах экрана.

*Практический критерий DE:* паттерн доступа (много ли тяжёлого чтения и батч‑записей) и риск положить источник.

- [ ] Почему нельзя бездумно ставить индексы на все поля?

Каждый индекс ускоряет конкретные чтения, но добавляет работу при изменении данных и усложняет планы (optimizer может выбрать не тот индекс). Много индексов ⇒ больше диска, дольше репликация/бэкапы, дороже администрирование.

*Подход инженера:* профилировать запросы, добавлять индекс под нагрузку, измерять latency и размер, пересматривать мертвые индексы.

- [ ] Что такое **первичный**, **уникальный** и **внешний** ключ?

**Первичный ключ (PK)** — главный идентификатор строки таблицы; обычно один на таблицу, NOT NULL, на него ссылаются как на «каноническую» строку. 

**Уникальный ключ (UK)** — тоже запрещает дубликаты по набору колонок, но таких ограничений может быть несколько; по NULL правила зависят от СУБД. 

**Внешний ключ (FK)** — ссылка на PK/UK «родителя»; включает ссылочную целостность и каскады (ON DELETE/UPDATE).

В DWH на измерениях часто вводят суррогатный ключ (**surrogate** — искусственный INT/BIGINT), а натуральный бизнес‑ключ хранят отдельно; в фактах — FK на измерения. В dbt проверки `relationships` как раз про FK‑семантику.

- [ ] Быстрее ли вставка в таблицу без первичного ключа, чем с ним?

Зачастую да: уникальность по PK поддерживается индексом; каждая вставка обновляет индексную структуру. Без PK/UK вставка может быть легче, но вы теряете гарантию уникальности и усложняете `MERGE`/`UPSERT`, дедуп и join по стабильному ключу.

В production «убрали PK ради скорости» часто означает технический долг и скрытые дубликаты.

- [ ] Чем **кластеризованный** индекс отличается от **некластеризованного**?

**Кластеризованный (Clustered Index)**

  - **Суть**: Определяет физический порядок хранения строк на диске. Данные отсортированы по ключу индекса.

  - **Хранение**: На конечных узлах (leaf nodes) хранятся сами данные (строки таблицы).

  - **Количество**: Может быть только один на таблицу (так как данные не могут быть отсортированы физически двумя способами одновременно). Обычно это Primary Key.

  - **Скорость**: Самый быстрый доступ к данным (Index-Only Scan не нужен, данные уже тут). Идеален для диапазонов (BETWEEN, >).

  - **Аналогия**: Телефонная книга (отсортирована по фамилиям, данные идут подряд).

**Некластеризованный (Non-Clustered Index)**

  - **Суть**: Отдельная структура, которая не влияет на физический порядок строк.

  - **Хранение**: На конечных узлах хранятся указатели (Row ID или ключ) на реальные строки данных.

  - **Количество**: Можно создать много (до 999 в SQL Server, тысячи в Postgres).

  - **Скорость**: Требует лишнего шага (Lookup) — сначала находим указатель в индексе, потом идем за данными в кучу (Heap) или кластеризованный индекс.

  - **Аналогия**: Предметный указатель в конце книги (Термин → Номер страницы).

*Как отвечать одной строкой:* Кластеризованный индекс сортирует данные физически и хранит сами строки на листьях; он может быть только один. Некластеризованный — это отдельная структура с указателями на данные, их может быть много, но чтение требует лишнего шага (Lookup).

- [ ] Как в общих чертах работает B-tree индекс?

Это сбалансированное дерево страниц фиксированного размера: от корня к листьям на каждом шаге выбирается ветвь по сравнению ключа с границами ключей в узле. Глубина O(log N) страниц ⇒ быстрые lookup/range scan. Листья часто связаны для обхода упорядоченного диапазона.

Вставка/сплиты страниц — причина, почему write-heavy нагрузка «пилит» диск и фрагментирует индекс.

- [ ] Почему после массового DELETE в PostgreSQL запросы могут стать медленнее?

При MVCC DELETE обычно помечает версии строк неживыми (dead), но место возвращается лениво через VACUUM. Появляются dead tuples, раздувается таблица и индексы, статистика становится неактуальной ⇒ планировщик может выбрать seq scan. Index-only scan страдает из‑за visibility map.

*Практика*: `VACUUM (ANALYZE)`, тяжёлые кейсы — `REINDEX`. В аналитике чаще перезапись партиции вместо массового DELETE всей истории.

- [ ] Когда оптимизатор PostgreSQL может не использовать индекс — и это нормально?

Если селективность низкая (индекс почти всё вернёт), дешевле последовательно прочитать таблицу. Если на колонке `lower(email)`, а индекс на `email`, индекс может не подойти без expression index. Маленькая таблица — накладные расходы random I/O по индексу могут превысить sequential read. Устаревшая статистика — план «угадывает» неправильно.

*Инженерный ответ:* `EXPLAIN (ANALYZE, BUFFERS)`, `ANALYZE`, исправить запрос/индекс под предикат.

- [ ] Чем **MPP** отличается от классического **SMP** (один сервер) и зачем это DE?

**SMP (Symmetric Multiprocessing)** на одном сервере — общая память и диски, параллелизм внутри машины; масштаб в основном вертикальный (больше RAM/CPU).

**MPP (Massively Parallel Processing)** — кластер узлов shared-nothing: данные шардированы, вычисления локальны, сеть соединяет узлы при shuffle/motion. Это путь к петабайтным сканам и тяжёлым `GROUP BY`.

Greenplum, Vertica, многие облачные DW — MPP. «Взяли OLTP Postgres и назвали его витриной» без моделирования распределения обычно не работает: узкое место — пересылки и skew по сегментам.

## DWH, моделирование и качество

- [ ] Зачем нужен **DWH**, если уже есть источники данных?

DWH нужен для консистентной аналитики: единые определения метрик, нормализованный доступ к данным из разных систем, историчность и воспроизводимость расчетов.

Без DWH каждая команда начинает считать «свои цифры», что быстро приводит к конфликтам в отчетности и росту техдолга.

- [ ] Чем отличаются **Data Lake**, **DWH** и **Data Lakehouse**?

**Data Lake**

  - Хранит: Сырые данные любого формата (структурированные, полуструктурированные, неструктурированные) — JSON, CSV, Parquet, изображения, логи.

  - Схема: Schema-on-Read — структура накладывается при чтении, а не при записи.

  - Пользователи: Data Scientists, инженеры для исследований, ML-прототипов.

  - Плюсы: Дёшево (object storage), гибко, масштабируется до экзабайт.

  - Минусы: Риск превратиться в «болото» без метаданных, нет гарантий качества, сложная оптимизация запросов.

  - Технологии: MinIO, S3, ADLS + Spark, Presto.

**Data Warehouse (DWH)**

  - Хранит: Только структурированные, очищенные, бизнес-данные для аналитики.

  - Схема: Schema-on-Write — строгая модель (звезда/снежинка) перед загрузкой.

  - Пользователи: Бизнес-аналитики, BI-отчёты, дашборды.

  - Плюсы: Высокая производительность агрегаций, гарантии качества, встроенная безопасность, зрелые инструменты.

  - Минусы: Дорого, менее гибко, сложно менять модель, плохо подходит для неструктурированных данных.

  - Технологии: ClickHouse, Snowflake, BigQuery, Redshift.

**Data Lakehouse**

  - Хранит: И сырые, и очищенные данные в едином слое на object storage.

  - Схема: Гибридная — Schema-on-Write для витрин + Schema-on-Read для исследований.

  - Пользователи: Все: от DS до бизнес-аналитиков в одной платформе.

  - Плюсы:

    ✅ Дешевизна Lake + транзакционность DWH (ACID через Iceberg/Delta/Hudi)

    ✅ Единый каталог данных, no ETL-copy между системами

    ✅ Time Travel, upserts, governance на одном слое

  - Минусы: Сложнее в настройке, моложе экосистема, требует экспертизы.

  - Технологии: Apache Iceberg / Delta Lake / Hudi + Spark/Trino + S3/MinIO.

*Как отвечать одной строкой на собеседовании:* Data Lake — дешёвое хранилище сырых данных любого формата с schema-on-read, идеален для исследований. DWH — структурированное, оптимизированное хранилище для BI с schema-on-write и высокими гарантиями. Lakehouse — гибридный подход: хранит всё в Lake, но добавляет транзакционность, индексы и governance как в DWH через форматы вроде Iceberg, объединяя плюсы обоих миров.

- [ ] ETL vs ELT — что выбрать и почему?

В современных облачных/MPP-стэках чаще выбирают ELT: сначала загрузка в хранилище, потом трансформации внутри него. Это быстрее для итераций и проще масштабировать.

ETL уместен, когда нужно жестко фильтровать/обогащать до хранилища, например из-за ограничений безопасности или стоимости хранения сырья.

- [ ] Где в типичном пайплайне «тяжелее» по ресурсам: классический ETL или ELT?

В ETL основная CPU/IO нагрузка часто на выделенном ETL-кластере (Spark, контейнеры) до попадания данных в хранилище; сеть и стоимость egress из источника тоже на счету. В ELT сначала грузят сырьё в lake/DWH, а тяжёлые SQL/dbt выполняются уже в MPP — растёт стоимость compute/лицензий СУБД и конкуренция с BI за слоты, но итерации по модели быстрее.

Выбор — про границы ответственности, безопасность данных и кто платит за пиковые агрегации.

- [ ] Зачем делить данные на raw/silver/gold?

Raw сохраняет «как пришло», silver приводит к стандарту и качеству, gold содержит бизнес-витрины. Это разделение упрощает отладку: можно локализовать проблему по слою.

Также это дает безопасный reprocessing: при изменении логики пересчитываешь silver/gold из raw без повторного запроса в источники.

- [ ] Какие слои хранения обычно выделяют в DWH/Lakehouse?

Минимальный практичный набор: landing/raw (как пришло), staging/bronze (техническая нормализация), silver/core (бизнес-сущности и качество), gold/marts (потребление BI/продуктом). В некоторых командах выделяют отдельный semantic layer для метрик.

Ключ — четкая ответственность слоя: где разрешено грязное сырье, где выполняется дедуп и контракты, где фиксируются витринные расчеты. Это уменьшает хаос при изменениях.

- [ ] Что такое **grain** факта и почему это критично?

**Grain** — это уровень детализации строки факта (например, «один заказ», «одна позиция в заказе»). Если grain выбран неявно или меняется по пути, итоговые метрики становятся недостоверными.

Сильные кандидаты всегда формулируют grain перед проектированием витрины и join с измерениями.

- [ ] Что такое **SCD Type 2** и когда он нужен?

SCD2 хранит историю изменений измерения через новые версии строк с периодом действия (`valid_from`/`valid_to`, current flag). Это нужно, когда важно видеть атрибут «как было на момент события».

Без SCD2 легко получить ошибку «пересчитали старые события по новым атрибутам» и испортить ретроспективную аналитику.

- [ ] Как в ежедневном ETL описать шаги SCD2 при сравнении снимка с текущей версией?

Сначала находят строки, где атрибут изменился относительно текущей открытой версии (`eff_to IS NULL`), и вставляют новую строку с новым значением и `eff_from = load_date`. Затем для затронутых сущностей закрывают предыдущую открытую запись (`eff_to = load_date - 1` или граница бизнес-правила). Если снимок совпал с текущим значением, ничего не делают — идемпотентность на день.

Важно в одной транзакции или с блокировкой по ключу определить порядок close/insert, чтобы не получить две «текущие» строки.

- [ ] Что такое data contract в DE?

Это формализованное соглашение между producer и consumer: схема, семантика полей, правила изменений, SLA свежести и качества. Контракт снижает риск неожиданных поломок downstream.

На практике это критично для масштабируемости команды: изменения в источнике становятся управляемыми, а не «сломалось ночью в проде».

- [ ] Что такое **Data Vault** и когда его используют?

**Data Vault** —  Методология моделирования данных для масштабируемых хранилищ, разработанная Дэном Линстедтом. Ориентирована на гибкость, аудируемость и параллельную загрузку.

**Три основных компонента**

**Hub (Хаб)**

  - Суть: Список уникальных бизнес-ключей (сущностей).

  - Примеры: hub_customer, hub_product, hub_order.

  - Колонки: hash_key (MD5/SHA бизнес-ключа), business_key, load_ts, record_source.

  - Важно: Никогда не удаляются, только добавляются новые ключи.

**Link (Связь)**

  - Суть: Соединяет хабы, отражает отношения «многие-ко-многим».

  - Примеры: link_order_customer, link_order_product.

  - Колонки: hash_key, hub_key_1, hub_key_2, load_ts, record_source.

  - Важно: Может хранить атрибуты самой связи (например, quantity в связи заказа и товара).

**Satellite (Спутник)**

  - Суть: Хранит описательные атрибуты и историю изменений (SCD Type 2).

  - Примеры: sat_customer_details, sat_product_pricing.

  - Колонки: hub/link_hash_key, attribute_1, valid_from, valid_to, load_ts.

  - Важно: Привязывается к хабу или линку, содержит временные метки для отслеживания истории.

**Ключевые принципы**

  - Аудируемость: Каждое поле имеет load_ts и record_source — всегда известно, откуда и когда пришли данные.

  - Масштабируемость: Новые источники добавляются как новые сателлиты без изменения существующих хабов/линков.

  - Параллельная загрузка: Хабы, линки и сателлиты можно грузить независимо друг от друга.

  - Историчность по умолчанию: Все изменения сохраняются, ничего не перезаписывается.
  
*Как отвечать одной строкой на собеседовании:* Data Vault — методология моделирования на основе хабов (бизнес-ключи), линков (связи) и сателлитов (атрибуты с историей). Используется в сложных enterprise-хранилищах с множеством источников, где критичны гибкость к изменениям, полная аудируемость и параллельная загрузка. Требует дополнительного слоя витрин для удобной аналитики, но даёт долгосрочную устойчивость архитектуры.

- [ ] Data Vault 2.0

**Data Vault 2.0**

  - Добавляет Business Vault — слой бизнес-правил, вычислений, деривативных атрибутов (отделяет «сырые» данные от бизнес-логики).

  - Добавляет Automation — стандартизированные паттерны для генерации ETL-кода.

  - Включает Governance — метаданные, качество данных, документация как часть методологии.

- [ ] Что менять в **Data Vault** при появлении нового источника той же бизнес-сущности?

Хабы по стабильному бизнес-ключу обычно расширяются новым `record_source`/потоком загрузки; новые атрибуты идут в новые или существующие satellites, новые связи — в links. Пересборка всей модели не требуется — это смысл DV.

Меняется ETL-парсинг и тесты на совместимость ключей и семантики полей.

- [ ] Как в **Data Vault** делят satellites и зачем?

Типичные оси: частота изменения атрибутов (hot/cold), разные системы-источники, требования безопасности (PII отдельно), тяжёлые поля/BLOB. Меньше версионных строк в «медленных» спутниках — дешевле хранение и быстрее загрузка.

На старте часто один satellite на сущность; дробят, когда появляется боль в размере, SLA загрузки или доступах.

- [ ] Что обязательно хранится в Hub Data Vault, а что туда класть нельзя?

В Hub кладут только стабильный бизнес-ключ (`BK`), технический ключ (`hash/surrogate`) и технические метаданные загрузки (`load_date`, `record_source`). Это «скелет сущности», без описательных атрибутов.

Имена, статусы, контакты и другие изменяемые поля в Hub не кладут — они живут в Satellites. Связи между сущностями — в Links.

- [ ] В чем разница подходов **Инмона** и **Кимбалла**?

**Подход Инмона** — сначала корпоративное нормализованное хранилище (top-down), потом витрины. 

**Подход Кимбалла** — сначала предметные витрины и размерные модели (bottom-up), интеграция через conformed dimensions.

На практике выбор зависит от зрелости команды и темпа изменений: Кимбалл быстрее дает бизнес-ценность, Инмон лучше дисциплинирует единое корпоративное ядро при сложных доменах.

- [ ] Что такое схемы **«звезда»** и **«снежинка»**?

**Звезда** — одна факт-таблица и денормализованные измерения, проще и быстрее для BI-запросов. 

**Снежинка** — измерения нормализованы на несколько таблиц, меньше дублирования, но больше join и сложнее запросы.

Для интервью хороший ответ: «для аналитики чаще звезда из-за простоты и скорости, снежинку применяют там, где важна строгая нормализация справочников».

- [ ] Что такое **Якорная модель (Anchor Modeling)** в контексте DWH?

**Что такое Anchor Modeling**

  - Определение: Методология гипернормализованного моделирования данных (1NF), разработанная Ларсом Рённбэком.

  - Суть: Разбивает данные на максимально мелкие сущности, где каждый атрибут хранится в отдельной таблице. Это позволяет изменять структуру без ALTER TABLE и избежать NULL значений.

**4 основных компонента**

**Anchor (Якорь)**

  - Суть: Бизнес-сущность (например, Customer).

  - Содержит: Только уникальный идентификатор (свой ID) и метаданные загрузки.

  - Аналогия: Корень дерева.

**Attribute (Атрибут)**

  - Суть: Свойство якоря (например, Customer_Name, Customer_Email).

  - Важно: Каждый атрибут — отдельная таблица. Это позволяет добавлять новые свойства без блокировки таблицы и изменения схемы.

  - История: Поддерживает темпоральность (изменения значений во времени) через valid_from / valid_to.

**Tie (Связь)**

  - Суть: Отношения между якорями (например, Customer_Order).

  - Содержит: Ссылки на ID якорей и, возможно, атрибуты самой связи.

**Knot (Узел)**

  - Суть: Переиспользуемый атрибут (например, Gender, Status).

  - Зачем: Если несколько якорей используют одно и то же перечисление, оно выносится в Knot, чтобы не дублировать данные.

**Главная фишка: «No ALTER TABLE»**

В классических базах добавление колонки блокирует таблицу. В Anchor Modeling добавление нового свойства — это просто CREATE TABLE нового атрибута и вставка данных. Схема «расширяется», а не модифицируется.

**Сравнение с другими моделями**

**vs Star Schema (Kimball):** Star — это широкие таблицы (много колонок), удобные для BI, но сложные для изменения. Anchor — это тысячи узких таблиц (много строк), сложные для BI, но легкие для изменения.

**vs Data Vault:** Data Vault группирует атрибуты в Satellites (пачки полей). Anchor идет дальше и разбивает каждое поле на отдельную таблицу. Anchor — это еще более «сырой» и детализированный слой.

*Как отвечать одной строкой на собеседовании:* Anchor Modeling — это гипернормализованная методология, где каждая сущность (Anchor) разбивается на отдельные таблицы для каждого атрибута (Attribute) и связи (Tie). Это позволяет избежать NULL и добавлять новые поля без ALTER TABLE, что идеально для хранения сырых данных с нестабильной схемой. Однако для аналитики требуется сборка данных через множество JOIN'ов, поэтому Anchor обычно используют в Raw-слое, а для BI строят поверх витрины.

- [ ] Зачем **суррогатные** ключи?

**Суррогатный ключ** — технический стабильный идентификатор, не зависящий от изменений бизнес-поля. Он упрощает JOIN и SCD2, особенно когда натуральный ключ составной/грязный/нестабильный.

Важно понимать, что суррогат не заменяет бизнес-ключ, а дополняет его для надежной обработки.

- [ ] Какие **DQ-проверки** (Data Quality) считаются базовыми?

Минимум: not null, uniqueness, referential integrity, freshness и проверки объема/аномалий. Эти тесты должны быть встроены в пайплайн, а не запускаться вручную.

Также полезно явно определить «блокирующие» и «предупреждающие» проверки, чтобы не останавливать весь контур из-за некритичной проблемы.

- [ ] Что такое **Fact Table** и **Dimension Table**?

**Fact table** хранит события и измеримые метрики (сумма заказа, количество, стоимость, время события), обычно с внешними ключами на измерения. 

**Dimension table** хранит описательные атрибуты сущностей (клиент, продукт, география, календарь), по которым делают фильтрацию и разрезы.

*Практическая идея:* факты растут быстро и широкими объемами, измерения меняются медленнее и используются для контекста. Корректная связка fact + dimension — основа надежной аналитики.

- [ ] Что такое **Business Key** и чем он отличается от **Surrogate Key**?

**Business key** приходит из предметной области (например, `order_number`) и несет смысл для бизнеса, но может быть нестабильным или составным. 

**Surrogate key** — технический идентификатор без бизнес-смысла, стабильный для join и хранения истории.

В DWH обычно держат оба: бизнес-ключ для трассировки в источник и суррогатный ключ для производительности и контроля версионности.

- [ ] Что такое **SCD** Type 1/2/3 и где каждый применяют?

**Type 1** перезаписывает значение без истории, подходит когда прошлое не важно. 

**Type 2** хранит новые версии строк с периодом действия и нужен для анализа "на момент события". 

**Type 3** хранит ограниченную историю (обычно текущее + предыдущее значение) и используется реже.

В интервью достаточно показать критерий выбора: нужна ли полная историчность, как часто меняется атрибут и какая цена хранения/сложности запросов.

- [ ] Как реализовать SCD Type 2 в SQL или Spark?

Логика одинаковая: детектировать изменения по business key + hash атрибутов, закрыть текущую версию (`valid_to`, `is_current=false`) и вставить новую (`valid_from`, `is_current=true`). В SQL это часто `MERGE`; в Spark — staged DataFrame + merge/overwrite по партиции.

*Важно обеспечить идемпотентность и детерминизм, чтобы повторный запуск не плодил лишние версии и не ломал интервалы действия.*

- [ ] Что такое Data Mesh и Data Fabric? Чем отличаются от классического DWH?

**Data Mesh** — организационный подход: доменные команды владеют своими data products с контрактами и SLA. 

**Data Fabric** — технологический слой унификации доступа/интеграции/metadata поверх разнородных платформ.

Классический DWH более централизован по модели и ownership. Mesh/Fabric не отменяют DWH, а меняют распределение ответственности и способы интеграции.

- [ ] Зачем отдельный core/silver слой между raw и витринами и когда можно без него?

Core — единая нормализованная зона качества: дедуп, контракты, историчность, общие сущности; 

витрины (gold) строятся от него и не дублируют «грязную» логику. Без core при нескольких командах и витринах быстро расходятся метрики и растёт техдолг.

Отказ ускоряет старт и экономит место на маленьких проектах с одним источником; при многих потребителях и compliance core почти всегда окупается.

- [ ] Как в SCD Type 2 отразить удаление записи в источнике?

Обычно не физически стирают строку в измерении, а делают soft delete: новая версия с `is_deleted=1` или статусом, закрывают предыдущую `valid_to`, в отчётах берут `is_current=1` и не удалённые. Так сохраняется аудит и корректный join к фактам на дату.

Альтернатива — отдельная таблица tombstone или обработка delete-событий из CDC; важно заранее договориться с BI, как показывать «исчезнувших» клиентов.

## Data Lake, Ingestion, CDC

- [ ] Что такое **Ingestion**?

**Ingestion** — это процесс доставки данных из источников в хранилище. Я различаю batch (по расписанию, для больших объёмов) и streaming (по событию, для low-latency), а также pull (тянем сами) и push (источник шлёт). Ключевые требования: идемпотентность, обработка схемы (schema evolution), надёжность (ретраи, DLQ) и мониторинг задержки. В проекте использую Airflow + Spark для batch, Debezium + Kafka для CDC-стриминга, с сохранением сырых данных в Landing Zone для возможности перезапуска.

- [ ] Что такое **Watermark**?

**Watermark** — это механизм в стриминге, который отслеживает прогресс времени и определяет, когда можно безопасно агрегировать данные, учитывая допустимую задержку для опоздавших событий. Например, watermark = 15:05 означает, что события раньше 15:05, скорее всего, уже пришли. Если событие приходит с event_time < watermark, оно считается late data и обрабатывается отдельно или отбрасывается. В Spark это настраивается через .withWatermark("event_time", "10 minutes"), в Flink — через assignTimestampsAndWatermarks. Ключевой компромисс: большой watermark повышает точность, но увеличивает задержку результатов.

- [ ] Как забирать данные из OLTP и не положить источник?

OLTP-база обслуживает продукт; тяжёлый `SELECT *` может сломать p95 latency транзакций. Типичные меры:

- Читать с реплики для отчётов (read replica), если она поддерживается, а не с мастера.
- Ограничивать частоту, размер скана, число параллельных воркеров; использовать батчи по диапазону ключа/партиции.
- По возможности не делать каждую ночь полный скан огромной таблицы, а использовать CDC (Change Data Capture — чтение журнала WAL/binlog) или водяной знак/инкремент по надёжной колонке.
- Ставить timeout/resource group совместно с DBA; понимать MVCC/длинные транзакции (undo может раздуться из-за вашей выгрузки).

Связать с бизнесом: «SLA продукта важнее, чем SLA нашей витрины минутами раньше». Частый паттерн: единоразовый snapshot (initial load), дальше CDC или инкремент.

- [ ] Что делать, если в источнике нет надежного уникального ключа?

Варианты (часто комбинируют):

- Составной бизнес-ключ из полей, которые вместе уникальны в домене.
- Генерация суррогатного ID уже на ingestion, если нужна ссылочность.
- Стабильный hash набора столбца (row hash) для дедупа — но это контракт: при смене полей меняется смысл.

Без понятной семантики ключа upsert/дедуп невозможны качественно; растут дубликаты, плавающие суммы.

- [ ] Чем CDC лучше инкремента по `updated_at`?

Инкремент по `WHERE updated_at > :watermark` прост в понимании, но:

- Одна запись может апдейтиться много раз между двумя ночными прогонками → вы можете видеть только последнее изменение (или наоборот, при ретраях — лишние перечитывания).
- Clock skew, NULL в штампе, забытый updated_at в триггерах.
- DELETE обычно не виден без отдельной политики.

CDC читает журнал транзакций и выдаёт insert/update/delete события (в зависимости от коннектора). Цена: replication slot, ретенция лога, схема событий, порядок и дубликаты на приёмнике.

- [ ] Что такое **small files problem** и как его решать?

Каждый файл в HDFS/S3 — это метаданные в NameNode/ листинги в object storage, открытие файла, планирование task в Spark. Миллионы крошечных файлов ⇒ планировщик тонет, merge в движке неэффективен.

Что делать: батчить запись (coalesce/правильное `spark.sql.files.maxPartitionBytes`), использовать Compaction (OPTIMIZE в Delta, expire_snapshots/rewrite в Iceberg — по документации инструментов), контролировать число партиций/не дробить день на сотню файлов без причины.

- [ ] Что такое **Parquet** и **ORC** и почему они лучше **CSV/JSON** для аналитики?

CSV/JSON обычно строка за строкой в хранении: дорого читать все столбцы, слабая типизация, компрессия менее эффективна, чем у бинарных колоночных форматов.

Parquet и ORC — колоночные форматы: читают только нужные колонки, хранят статистику (min/max) по row groups/stripes для predicate pushdown на уровне файла, поддерживают кодеки (Snappy, ZSTD). Spark и Trino хорошо используют такой columnar путь.

- [ ] Что такое **Partitioning** и **Bucketing** в файловых данных?

Hive-style partitioning хранит подкаталоги `dt=2025-05-06`, чтобы отрезать целые папки по `WHERE` (это coarse partition pruning).

Bucketing делит каждую таблицу/папку на `N` файлов по `hash(bucket_key) mod N`; полезно, когда нужно повторять JOIN без глобального shuffle между запусками, если обе таблицы согласованно захэшированы.

- [ ] Что такое **Predicate Pushdown**?

**Predicate pushdown** — оптимизатор проталкивает условие `WHERE` ниже по стеку: к сканированию файла (Parquet читает меньше row groups), к коннектору JDBC (фильтр в запрос к источнику, где это поддерживается), к Kafka где возможно. Итог: меньше сетевых байт, быстрее job.

Разница между «написали SQL» и «сделали дорого» часто именно здесь — проверяйте план/логику коннектора.

- [ ] Что такое объектное хранилище (S3/MinIO) и чем оно отличается от «диска на сервере»?

Объектное хранилище (S3/MinIO) — это система, где данные хранятся как объекты (файл + метаданные) в плоской структуре бакетов с доступом по HTTP API. В отличие от блочного диска (EBS), который монтируется к одному серверу и позволяет random read/write, объектное хранилище масштабируется горизонтально до экзабайт, дешевле, но не поддерживает изменение части файла — только полную перезапись. Идеально для Data Lake, бэкапов и медиа, но не подходит для БД из-за высокой latency. 

В проекте использую S3/MinIO как raw layer с партиционированием по дате и lifecycle policies для автоархивации

- [ ] Что такое бакет, ключ объекта и pre-signed URL в S3-совместимых системах?

**Bucket (Бакет)** — Контейнер верхнего уровня для хранения объектов. Аналог «корневой папки» или «базы данных» в мире файлов.

**Object Key (Ключ объекта)** — Уникальный идентификатор объекта внутри бакета. Полный «путь» к файлу.

**Pre-signed URL (Временная ссылка)** — Сгенерированная на основе ваших креденшалов временная ссылка на приватный объект, которая даёт доступ любому, у кого есть эта ссылка, без необходимости иметь AWS-ключи.

- [ ] Какие ограничения объектного хранилища важно помнить при аналитике?

**Объектное хранилище** — не «медленный диск», а другая парадигма. Не пытайтесь эмулировать файловую систему — проектируйте пайплайны под immutable append-only паттерн.

**Мелкие файлы** — главный враг. Один файл на 1 ГБ обрабатывается в 1000 раз быстрее, чем 1000 файлов по 1 МБ (из-за overhead на задачу и метаданные).

**Метаданные** — ваш друг. Правильное партиционирование (date=, region=) и использование каталогов (Glue/Hive) превращают S3 из «свалки» в эффективный аналитический слой.

**Сочетайте с вычислительными движками.** S3 + Spark/Presto/ClickHouse — это сила. S3 сам по себе — просто хранилище.

*Как отвечать одной строкой на собеседовании:* Ключевые ограничения объектного хранилища для аналитики: нет random access (только полная перезапись), высокая latency (не для БД), проблема мелких файлов (тормозит листинг и Spark), отсутствие POSIX (сложные паттерны записи), плата за операции (мелкие запросы дороги). Решаю это: пишу крупные колоночные файлы (Parquet >128MB), использую партиционирование и predicate pushdown, применяю транзакционные форматы (Iceberg) для upsert, и кэширую метаданные для ускорения листинга.

- [ ] Как перенести очень большую таблицу из OLTP в lake/DWH, не положив источник (идея high-level)?

Разбить на потоки: «лёгкие» колонки батчами по партициям/диапазонам id с параллельными воркерами, тяжёлые BLOB/документы — отдельным медленным пайплайном по ключу. Писать сначала в staging в lake (Parquet), сверять counts/checksums, потом атомарно публиковать в целевой слой.

Инкремент по окну или CDC после первичного снимка; throttling и read replica обязательны.

- [ ] Что делать, если при инкрементальной выгрузке в источнике постоянно обновляют уже прочитанные строки?

Стандартный инкремент по `updated_at` между чанками может дать фантомы и рассинхрон. Нужен согласованный снимок: изоляция `REPEATABLE READ`/`SNAPSHOT` на сессию выгрузки, или CDC из WAL, или пересчёт затронутых партиций.

Цена — нагрузка на undo/MVCC и риск длинных транзакций; SLA выгрузки может сдвинуться, зато данные не «рвутся» между чанками.

- [ ] Какие альтернативы инкременту по полю `updated_at` для поставки изменений?

CDC (Debezium и др.), триггеры/очередь изменений, логическая репликация, сравнение снимков (diff двух extract), event sourcing в Kafka. Они ловят delete и промежуточные правки, которые watermark не видит.

Выбор зависит от допустимой нагрузки на источник и зрелости команды в стриминге.

- [ ] Три источника: у двух есть надёжный timestamp, у третьего нет. Как забирать инкремент каждый день?

Для двух — watermark или CDC. Для третьего — полный срез в staging с хэшем строки и сравнение с вчерашним снимком, или CDC если доступен журнал, или ночной snapshot в файл и diff по ключу.

Часто третий источник становится кандидатом на договорённость с владельцем о добавлении `updated_at` или на CDC.

- [ ] Как работает Debezium при шардированной базе?

Нужен отдельный коннектор (или логический слот) на каждый шард/инстанс с собственным WAL/binlog; в downstream ключи дополняют shard id, чтобы глобальная уникальность событий не коллизировала. Потребители либо мержат потоки, либо пишут в шардированный sink с тем же ключом.

Операционно сложнее: ретенция логов и координация схемы на всех шардах должны быть согласованы.

- [ ] Чем сильнее сжатие Parquet и что за это платим?

Сильнее codec/level — меньше байт на диске и в сети, но больше CPU на чтение/запись. Для аналитики обычно выигрывает экономия I/O; на CPU-bound кластере иногда откатываются к Snappy/LZ4.

Выбор — профилировать на своих данных, а не копировать «лучший codec из интернета».

- [ ] Как ловить пропуски данных после обрыва сети между источником и приёмником?

Сверка объёмов и контрольных сумм по партиции/батчу, идемпотентная перезагрузка упавшего диапазона, запись сначала во временную зону и commit только после успеха всей пачки.

Без reconciliation «тихие» потери всплывают только в отчётах бизнеса.

## AIRFLOW

- [ ] Что такое **DAG** и **Task** в Airflow?

**DAG (Directed Acyclic Graph — направленный ациклический граф)** в Apache Airflow — это описание пайплайна: какие задачи (tasks) от каких зависят, в каком расписании или по какому триггеру запускается DAG Run, какие параметры по умолчанию, теги, владелец. 

**Task** — конкретный шаг графа: экземпляр Operator (например PythonOperator), Sensor или TaskFlow-функция; у task есть состояния (success/failed/up_for_retry), ретраи, логи, XCom.

Ключевой тезис: Airflow оркестрирует внешнюю работу (Spark-submit, SQL, bash, API), а тяжёлые данные не «живут» внутри Airflow — туда кладут ссылки и метаданные, а не гигабайты payload.

- [ ] Что происходит от появления `.py` в `dags_folder` до отображения DAG в UI?

Процесс `DagFileProcessor` (часть scheduler): периодически читает `.py`, импортирует модуль на верхнем уровне, ищет объекты `DAG`, сериализует их описание в metadata DB (PostgreSQL чаще всего). Webserver не парсит DAG-файлы при каждом клике — он читает уже сохранённое в БД, поэтому битый import → DAG пропадает из UI целиком.

Практические следствия: не кладите тяжёлый I/O и сетевые вызовы на импорт модуля; не вызывайте `Variable.get()` на глобальном уровне без нужды — это бьёт по БД на каждом parse и тормозит весь scheduler.

- [ ] Какие основные операторы нужно знать?

Минимум для DE: 
`PythonOperator`/`BashOperator` — обёртка над скриптом; 

`EmptyOperator` — визуальный якорь/синхронизация веток; 

SQL-операторы (`PostgresOperator`, универсальный `SQLExecuteQueryOperator` в новых версиях) — прозрачный DDL/DML в git/review; 

`TriggerDagRunOperator` — explicit downstream; 

`ExternalTaskSensor` — ждать конкретную задачу другого DAG; 

`KubernetesPodOperator` — контейнер с зафиксированными зависимостями; deferrable сенсоры — ждать внешнее событие без удержания воркера.

Названия меняются между версиями Airflow, но идея одна: Operator задаёт шаблон действия, Executor решит где оно выполнится.

- [ ] Что такое **Sensor** и чем отличаются poke/reschedule/deferrable?

**Sensor** — Task, который крутится, пока внешнее условие не станет истинным (файл появился, партиция прогрузилась, upstream задача успешна). 

Режимы: `poke=True` (mode=`poke`) — сенсор удерживает слот воркера между проверками (дорого при долгом ожидании). 

`mode='reschedule'` — между проверками слот воркера освобождается, сенсор встаёт в паузу (компромисс). Deferrable («async») сенсоры — ожидание уносится в `triggerer`, worker освобождается сразу (лучший масштаб при долгих ожиданиях).

В проде неправильный сенсор может исчерпать pool/воркеры; иногда выгоднее Dataset/event-driven связка вместо вечного sensor.

- [ ] Что такое **XCom** и почему его нельзя использовать для больших данных?

**XCom** — механизм обмена небольшими метаданными между тасками через metadata DB Airflow. Он подходит для id, путей, параметров, статусов.

Если передавать большие payload, metadata DB раздувается и Airflow деградирует. Большие данные должны лежать в S3/БД/файлах, а в XCom передается только ссылка.

- [ ] Чем **XCom** отличается от **Airflow Variables**?

**Variables** — глобальные конфиги (ключ/значение) в metadata DB, живут долго, подходят для путей и флагов окружения; читать их на верхнем уровне DAG-файла дорого из-за parse. 

**XCom** — временный обмен результатами между тасками одного или связанных прогонов, хранится привязанно к `dag_id`/`task_id`/`run_id`, по умолчанию с ограничением размера.

Правило: в Variable — статика конфигурации, в XCom — небольшие динамические артефакты шага (id партиции, счётчик, путь к промежуточному файлу).

- [ ] Как делать DAG идемпотентным?

Нужно явно привязывать логику к `logical_date`/партиции и писать шаги так, чтобы повторный запуск давал тот же результат: merge, overwrite partition, детерминированные ключи и фильтры.

*Частая ошибка* — использовать «текущее время» без контекста запуска. Тогда backfill/catchup дают другой результат, и отладка превращается в хаос.

- [ ] Как связать DAG-и между собой?

Есть несколько способов: datasets (событийная связка), `ExternalTaskSensor` (ждать конкретный task), `TriggerDagRunOperator` (явно запускать downstream). Выбор зависит от того, нужен ли pull- или push-подход.

*Хорошая практика* — избегать «временных зависимостей» типа «ждем +30 минут», если можно связать DAG по факту готовности данных.

- [ ] В чем разница **LocalExecutor** и **CeleryExecutor**?

**LocalExecutor** запускает параллельные процессы на одной машине — удобно для dev/малых стендов. 

**CeleryExecutor** распределяет задачи по воркерам через брокер (Redis/RabbitMQ), лучше подходит для production-масштаба.

На интервью стоит добавить, что **KubernetesExecutor** — еще один production-вариант с изоляцией задачи в отдельном pod.

- [ ] Что такое **catchup** и **backfill**?

**Catchup** — автоматический догон пропущенных интервалов по расписанию. 

**Backfill** — ручной пересчет диапазона дат, обычно через CLI/API.

Оба режима требуют идемпотентной логики. Если задачи не идемпотентны, получишь дубли и расхождения при восстановлении истории.

- [ ] Что такое **DAG factory** и зачем она в крупных DE-командах?

Это генерация множества однотипных DAG из конфигурации (YAML/JSON/таблица): цикл или фабричная функция создаёт объекты `DAG` с разными `params`, расписанием и списком таблиц. Так стандартизируют сотни однотипных загрузок и снижают копипаст.

*Риск* — общий код при ошибке ломает сразу группу DAG; нужны тесты парсинга и ограничение динамики.

- [ ] Чем **TaskGroup** отличается от **SubDAG** в Airflow?

**TaskGroup** — это логическая группировка задач в интерфейсе и коде внутри одного DAG без отдельного scheduler-контекста. 

**SubDAG** исторически создавал вложенный DAG со своей семантикой выполнения, из-за чего часто появлялись deadlock/сложности с пулами и он считается устаревшим подходом.

*Практический ответ на интервью:* использовать `TaskGroup` для структуры и читаемости, а не `SubDAG`.

- [ ] Для чего нужны **pools** и **max_active_runs**?

**Pools** ограничивают параллелизм по ресурсу (например, у внешнего API есть rate limit). 

**max_active_runs** ограничивает число одновременных прогонов конкретного DAG.

Это ключевые регуляторы стабильности. Без них легко перегрузить источники и утонуть в очередях задач.

- [ ] Как мониторить Airflow-пайплайн в production?

**Нужны три слоя:**

  - Алерты (on-failure, SLA miss)

  - Метрики (длительность, ретраи, очередь, success rate)

  - логи (детализация ошибки и контекст запуска)

Сильный ответ всегда включает не только «увидеть красное», но и «быстро локализовать причину».

- [ ] Как организовать зависимости между DAG-ами: ExternalTaskSensor vs TriggerDagRunOperator vs Assets/Datasets?

**ExternalTaskSensor**

  - Суть: Сенсор «ждет» завершения задачи (или всего DAG-а) в другой DAG-цепочке.

  - Как работает: Airflow периодически опрашивает базу данных метаданных (SELECT state FROM task_instance...).

  - Тип связи: Pull (Текущий DAG тянет состояние другого).

*Плюсы:*

  - Простой в настройке.

  - Гарантирует, что текущий DAG не начнется, пока другой не закончится.

*Минусы:*

  - Жесткая связность: Если в upstream DAG переименовать задачу (task_id), downstream DAG сломается.

  - Нагрузка на БД: Частый polling (опрос) может нагружать Metadata DB.

  - Слепота: Не передает данные, только статус успеха/неудачи.

**TriggerDagRunOperator**

  - Суть: Текущий DAG активно запускает другой DAG.

  - Как работает: Создает новый DagRun для другого DAG по завершении задачи.

  - Тип связи: Push (Текущий DAG толкает триггер другому).

*Плюсы*:

  - Передача параметров: Можно передать conf (JSON) в запускаемый DAG (например, дату или ID клиента).

  - Динамика: Позволяет запускать один и тот же DAG с разными параметрами.

*Минусы*:

  - Разрыв графа: В UI Airflow может быть сложно отследить полный путь выполнения, если DAG-ов много.

  - Сложность Backfill: Если перезапустить весь процесс, нужно аккуратно обрабатывать уже запущенные инстансы.

**Assets / Datasets (Airflow 2.4+ )**

  - Суть: Расписание привязано не ко времени, а к факту обновления данных.

  - Как работает:

  - DAG-источник помечает объект как обновленный: @asset("s3://bucket/data.csv").

  - DAG-потребитель запускается автоматически при обновлении этого объекта: schedule=[Dataset("s3://bucket/data.csv")].

  - Тип связи: Data-Driven (Слабая связность).

*Плюсы*:

  - Полная изоляция: DAG-потребитель не знает ID DAG-источника или задач. Ему важны только данные.

  - Many-to-Many: Один датасет может обновляться несколькими DAG-ами и запускать несколько зависимых DAG-ов.

  - Автоматический Backfill: Airflow понимает историю обновлений датасетов и может корректно пересчитать зависящие DAG-и.

*Минусы*:

  - Требует версии Airflow 2.4+.

  - Нужно внедрять понятие Asset в код обоих DAG-ов.

*Как отвечать одной строкой:* Для простых связок я использую ExternalTaskSensor, но он создает жесткие зависимости по именам задач. Если нужно передать параметры, использую TriggerDagRunOperator. Однако золотой стандарт (Airflow 2.4+) — это Assets/Datasets. Это позволяет реализовать Data-Driven архитектуру, где DAG-и изолированы друг от друга и реагируют на появление данных, а не на статус задач, что решает проблемы с переименованием и упрощает бэкфилл

- [ ] В чем разница между schedule interval (cron) и timedelta в расписании?

Cron выражает календарную логику (например, каждый день в 03:00), timedelta — относительный интервал между запусками. Cron удобнее для бизнес-расписаний, timedelta — для технических циклов.

На интервью полезно сказать, что важно понимать `data interval/logical date`, а не только "когда стартовал run".

- [ ] Как передавать параметры в DAG при ручном запуске?

Обычно используют `dag_run.conf` и параметры (`params`) в шаблонах/коде задачи. Это позволяет запускать один и тот же DAG для разных дат, таблиц или режимов.

Критично валидировать входные параметры, чтобы ручной запуск не испортил прод-данные.

- [ ] Когда ретраи в Airflow вредны?

Ретраи вредны при логических ошибках кода, неверных входных данных и неидемпотентной записи — они только умножают ущерб. Ретраи полезны для временных сетевых/инфраструктурных сбоев.

Хороший ответ: сначала классифицировать тип ошибки, потом решать retry или fail-fast с алертом.

- [ ] Что такое Dataset в Airflow и зачем он?

Dataset (в терминах data-aware scheduling) — описание логического или физического артефакта данных (URI таблицы, путь в lake и т.д.), с которым DAG может быть связан: при обновлении датасета планировщик может запустить downstream-DAG без жёсткой привязки ко времени.

Это сдвиг от «запуск в 03:00» к «запуск, когда данные готовы». На собеседовании полезно сопоставить с Sensors и TriggerDagRun: Dataset даёт декларативную связь по факту данных.

- [ ] В чём разница между Operator, Task, Hook и Sensor?

**Operator** — класс шаблона действия (например, запуск SQL). 

**Task** — конкретный экземпляр оператора внутри DAG run с состоянием, ретраями и XCom. 

**Hook** — низкоуровневая обёртка подключения к системе (БД, S3), которую переиспользуют операторы. 

**Sensor** — подвид оператора, который ждёт внешнее условие.

*Короткая формула:* Hook — «как подключиться», Operator — «что сделать», Task — «один запуск этого в прогоне», Sensor — «ждать условие».

- [ ] Какие типы операторов ты бы назвал на собеседовании и зачем они?

Базовый набор: 

`PythonOperator`/`BashOperator` для скриптов

SQL-операторы для прозрачных DDL/DML

`KubernetesPodOperator` для изолированного окружения

`TriggerDagRunOperator` для цепочки DAG-ов, сенсоры для ожидания файла или внешней задачи.

Важно объяснить выбор: например, Pod для воспроизводимых зависимостей, SQL-оператор для аудита того, что именно выполнилось в БД.

- [ ] Зачем Airflow Variables и почему не стоит звать `Variable.get()` на верхнем уровне DAG-файла?

**Variables** — ключ-значение в metadata DB для конфигов (пути, флаги). `Variable.get()` на уровне модуля выполняется при каждом parse DAG и бьёт по БД и latency парсинга.

*Паттерн*: читать переменные внутри callable таски или использовать шаблоны/`@task`, либо env/конфиги образа для статики.

- [ ] Чем отличаются Workers в Airflow от Scheduler и как «освободить» ресурс после таски?

Workers (Celery/K8s) исполняют task process; scheduler только планирует. На KubernetesExecutor под под задачу поднимается и гаснет — ресурс освобождается автоматически; на Celery воркеры долгоживущие — масштабирование через пул воркеров/KEDA.

Смысл вопроса на собеседовании — понимание, где реально крутится пользовательский код.

- [ ] Как не запускать новый прогон DAG, пока предыдущий упал (`depends_on_past` vs `max_active_runs`)?

`max_active_runs=1` лишь ограничивает параллельные прогоны, но не связывает успех вчера с запуском сегодня. 

`depends_on_past=True` на тасках заставляет сегодняшний инстанс ждать успешного вчерашнего для той же таски — это ближе к «не идти дальше после провала».

Иногда вместо этого явно проверяют качество данных или используют Dataset-триггеры после успешной публикации артефакта.

- [ ] DAG в Airflow стал стабильно выполняться дольше: в каком порядке разбирать причину?

Сначала объём входа и аномалии по датам, затем ресурсы кластера и очередь/пулы, логи таски и внешней системы (БД locks), не было ли деплоя, затем план запроса (`EXPLAIN`) и партиции/индексы.

Такой чеклист показывает системный подход, а не угадывание «наверное Spark».

## KAFKA

- [ ] Что такое **topic**, **partition** и **offset**?

**Topic** — Логическое имя потока данных (категория/канал), в который продюсеры пишут, а консюмеры читают.

Особенности: Не хранит данные физически, а лишь группирует их. Делится на партиции для горизонтального масштабирования.

**Partition** — Физическое разделение топика на независимые, упорядоченные последовательности записей.

Ключевые свойства:

✅ Упорядоченность: Строгий порядок событий только внутри одной партиции.

✅ Append-only: Данные нельзя изменить или удалить (только expire по retention policy).

✅ Распределение: Каждая партиция хранится на одном брокере (лидер) + реплики на других для отказоустойчивости.

**Offset** — Монотонно возрастающий порядковый номер сообщения внутри одной партиции (0, 1, 2, ...).

Как работает:
  - Не является глобально уникальным (в разных партициях номера повторяются).
  - Консюмер запоминает свой текущий offset, чтобы при рестарте продолжить чтение с того же места.
  - Коммиты offset'ов хранятся во внутреннем системном топике Kafka (__consumer_offsets).

*Как отвечать одной строкой на собеседовании:* Topic — логический поток данных. Partition — физическое упорядоченное разделение топика для параллельной обработки, где каждая партиция живёт на одном брокере. Offset — монотонно растущий номер сообщения внутри партиции, по которому консюмер отслеживает позицию чтения. Вместе они дают масштабируемость, гарантированный порядок в рамках ключа и надёжное восстановление состояния.

*Ключевой инсайт для продвинутого ответа:*

  - **Порядок гарантируется только внутри партиции.** Глобального порядка в топике нет. Чтобы события одного ключа (например, user_id) шли подряд, используйте его как partition key — тогда Kafka направит их в одну партицию.
  - **Количество партиций фиксируется при создании.** Увеличить можно, уменьшить — нельзя (без пересоздания топика и потери данных). Планируйте пиковую нагрузку заранее.
  - **Offset ≠ бизнес-ID.** Offset — это позиция в логе, генерируется брокером автоматически. Для идемпотентности используйте бизнес-ключи или идемпотентного продюсера.

- [ ] Как устроены Producer и Consumer в Kafka простыми словами?

**Producer (Продюсер / Производитель)**

  - Кто это: Приложение, которое отправляет данные в Kafka.

*Как работает (3 шага):*

  - Выбор топика и партиции:

  - Продюсер указывает topic, куда писать.

  - Если задан key (например, user_id), Kafka хэширует его и направляет сообщение в одну и ту же партицию (гарантия порядка).

  - Если ключа нет — работает Round-Robin или Sticky Partitioning (равномерное распределение).

*Батчинг (оптимизация):*

Сообщения не летят по одному. Продюсер копирует их в буфер и отправляет пачками (по 16-64 КБ) раз в несколько миллисекунд. Это резко повышает пропускную способность.

*Подтверждение (Ack):*

Продюсер ждёт ответа от лидера партиции:

  - acks=0 — «отправил и забыл» (быстро, но можно потерять).

  - acks=1 — лидер записал (баланс).

  - acks=all — лидер и все реплики записали (надёжно, но медленнее).

**Consumer (Консьюмер / Потребитель)**

  - Кто это: Приложение, которое читает данные из Kafka.

*Как работает (3 шага):*

  - Подписка и группы:

  - Консьюмер подписывается на topic.

  - Он входит в Consumer Group — логическую группу, которая совместно потребляет данные.

*Ребалансировка (Partition Assignment):*

  - Kafka автоматически распределяет партиции между консьюмерами группы.

*Важно:* Одна партиция может быть назначена только одному консьюмеру в группе (но консьюмер может читать много партиций).

*Чтение и коммит:*

  - Консьюмер читает сообщения последовательно по offset.

  - После обработки он отправляет коммит — подтверждение, что сообщение обработано.

  - При перезапуске консьюмер продолжит с последнего закоммиченного offset.

*Как отвечать одной строкой на собеседовании:* Producer отправляет данные в топик, используя ключ для гарантии порядка (в рамках партиции) и батчинг для производительности, настраивая надёжность через acks. Consumer читает данные в составе группы, где партиции распределяются между участниками, и отслеживает прогресс через коммит оффсетов. Ключевое: одна партиция в группе читается только одним консьюмером, что позволяет горизонтально масштабировать обработку, просто добавляя новые инстансы.

*Ключевой инсайт для продвинутого ответа:*

  - **Ключ — это порядок.** Если вам важен строгий порядок событий (например, «создан → обновлён → удалён»), всегда используйте один и тот же бизнес-ключ. Тогда все события попадут в одну партицию и будут прочитаны по очереди.
  - **Группа — это параллелизм.** Один консьюмер = одна нить чтения. Хотите ускориться? Увеличьте количество партиций при создании топика и добавьте консьюмеров в группу.
  - **Коммит — это ответственность.** Если вы делаете auto.commit=true, но приложение падает после коммита, но до реальной обработки — данные потеряны. Для критичных систем используйте ручные коммиты после успешной бизнес-обработки.

- [ ] Что такое **log retention** в Kafka и какой типичный дефолт?

**Retention** — сколько времени (`log.retention.ms`) и/или сколько байт (`log.retention.bytes`) Kafka держит старые log segments. Для `delete` топиков сегменты удаляются. Для log-compacted топиков — отдельная модель (compaction: по ключу остаются актуальные значения, что полезно для changelog тем в Kafka Streams, но это не полный исторический архив без проектирования).

*Ответ одной строкой:* Log retention — это время жизни данных в топике. По дефолту 7 дней (log.retention.hours=168), но можно настроить удаление по размеру или использовать log compaction для хранения только последних значений по ключу

*Инсайт:*

**Компромисс**: Долгий ретеншн = возможность перечитать историю, но больше дисков. Короткий = экономия, но риск потери при простое консьюмера.

**Compaction ≠ Retention**: Compaction удаляет дубликаты по ключу (оставляет последнее значение), retention удаляет по времени. Можно использовать вместе.

- [ ] Что такое consumer group?

**Consumer group** — это логическое объединение консьюмеров, которые совместно читают один или несколько топиков Kafka для параллельной обработки данных.

**Ключевые принципы:**

  - *Параллелизм* - Партиции топика распределяются между консьюмерами группы. Одна партиция → только одному консьюмеру в группе.

  - *Масштабирование* - Чтобы читать быстрее, добавьте консьюмеров в группу (но не больше, чем партиций в топике).

  - *Изоляция* - Разные группы читают один и тот же топик независимо, каждая со своего оффсета.

  - *Ребалансировка* - При добавлении/удалении консьюмера или падении узла, Kafka автоматически перераспределяет партиции.

  *Ответ одной строкой на собеседовании:* Consumer Group — это механизм параллельного потребления: партиции топика распределяются между консьюмерами группы, одна партиция — одному консьюмеру. Это позволяет горизонтально масштабировать обработку, просто добавляя инстансы, при этом разные группы читают данные независимо друг от друга

- [ ] Какие есть семантики доставки в Kafka?

В Kafka существуют три семантики доставки, которые определяют гарантии того, сколько раз сообщение будет доставлено потребителю.

**At-most-once**: - *Не более одного раза*

*Как работает:*
  - Producer отправляет сообщение и не ждёт подтверждения от брокера
  - Consumer читает сообщение и сразу коммитит offset, даже если не успел обработать
  - Если потребитель упал после коммита, но до обработки — сообщение потеряно 

**At-least-once**: - *Хотя бы один раз*

*Как работает:*
  - Producer ждёт подтверждения от брокера (ack)
  - Consumer читает сообщение, обрабатывает, и только потом коммитит offset
  - Если потребитель упал до коммита — при рестарте он прочитает сообщение снова

*Как бороться с дублями:*
  - Идемпотентный потребитель: Использовать уникальные ключи и проверять, обработано ли уже сообщение
  - Transactional Outbox Pattern: Записывать результат обработки и offset в одну транзакцию
  - Deduplication на уровне БД: INSERT IGNORE или ON CONFLICT DO NOTHING

**Exactly-once** - *Ровно один раз*

*Как работает:*
Kafka реализует это через транзакции и идемпотентных продюсеров.

*Компоненты:*

*Идемпотентный Producer (enable.idempotence=true):*
  - Каждому сообщению присваивается уникальный PID (Producer ID) и sequence number
  - Брокер отклоняет дубликаты по этим идентификаторам

*Транзакции:*
  - Producer начинает транзакцию
  - Пишет сообщения в несколько топиков/партиций
  - Коммитит транзакцию (все сообщения становятся видимыми) или абортирует (все отменяются)

*Isolation Level на Consumer:*
  - read_committed — читать только закоммиченные сообщения
  - read_uncommitted — читать все (по умолчанию)

- [ ] Что такое **acks** в Kafka и как связаны синхронная/асинхронная отправка?

`acks=0` — не ждать подтверждения записи (min latency, risky). 

`acks=1` — записалось у leader (компромисс). 

`acks=all` — подтвердили все ISR‑реплики (safest, больше чувствительность к узкому месту среди синхронных реплик).

*Ответ одной строкой на собеседовании:* Acks — это уровень подтверждения записи: 0 (не ждём), 1 (ждём лидера), all (ждём все ISR-реплики). Асинхронная отправка (send + callback) даёт высокий throughput, синхронная (send().get()) — проще в коде, но медленнее. На практике использую async + acks=all для критичных данных (надёжно и быстро) и async + acks=1 для обычных задач.

**Инсайт:**
  - **acks ≠ гарантия доставки.** Даже acks=all не защищает от потери, если упали все реплики одновременно. Для 100% надёжности нужен идемпотентный продюсер + транзакции.
  - **enable.idempotence=true** автоматически ставит acks=all и защищает от дублей при ретраях — включайте эту настройку по умолчанию.
  - **Асинхронность — это батчинг.** Кафка эффективно сжимает и отправляет пачки сообщений только при асинхронном вызове. Синхронная отправка убивает эту оптимизацию.

- [ ] Структура сообщения в Kafka

**Key** - Опциональный байтовый массив. Используется для выбора партиции и гарантии порядка событий одного ключа.

**Value** - Полезная нагрузка (тело сообщения). Часто в бинарном формате (Avro/Protobuf) или JSON.

**Timestamp** - Время события. Бывает CreateTime (когда создал продюсер) или LogAppendTime (когда брокер записал).

**Headers** - Массив пар key-value (строки). Для метаданных: correlation_id, tenant_id, source. Не влияют на сериализацию тела.

**Offset** - Позиция сообщения в партиции (присваивается брокером, не передаётся продюсером).

*Ответ одной строкой на собеседовании:* Сообщение в Kafka состоит из опционального Key (для партиционирования), Value (полезные данные), Timestamp, Headers (метаданные) и Offset (позиция в логе, присваивается брокером). При использовании Avro физически в топике хранится [Magic Byte] + [Schema ID] + [Бинарные данные], а полный набор записей упаковывается в RecordBatch для эффективной записи.

- [ ] Что нужно знать про схему сообщений в Kafka?

Зачем нужна схема: 

✅ **Валидация:** Отклоняет сообщения с неверной структурой на входе.

✅ **Эволюция:** Позволяет безопасно менять формат (добавлять поля) без поломки старых консьюмеров.

✅ **Сериализация:** Бинарные форматы (Avro/Protobuf) компактнее и быстрее JSON.

✅ **Документация:** Схема — это контракт, понятный и продюсерам, и консьюмерам.

Schema Registry — центральный компонент, отдельный сервис (обычно от Confluent), который хранит версии схем и проверяет совместимость.

**Как работает:**

1. Продюсер при отправке проверяет схему в реестре, получает schema_id.

2. Сообщение в Kafka содержит: [schema_id] + [бинарные данные].

3. Консьюмер читает schema_id, запрашивает схему из реестра и десериализует данные.

**Преимущество:** В топике хранятся только данные + 4 байта на ID, а не вся схема в каждом сообщении.

**Форматы схем:**

*Avro* - Компактный бинарный, богатая экосистема, стандарт де-факто

*Protobuf* - Очень компактный, поддерживает больше типов, популярен в gRPC

*JSON Schema* -  Человекочитаемый, не требует компиляции

**Режимы совместимости (Compatibility Modes):** 

*BACKWARD (дефолт)* - Новые консьюмеры читают старые данные

*FORWARD* - Старые консьюмеры читают новые данные

*FULL* - И старые, и новые консьюмеры работают с любыми данными

*NONE* - Любые изменения

*Ответ одной строкой на собеседовании:* Для управления схемами в Kafka использую Schema Registry с форматом Avro. Это даёт компактную сериализацию, валидацию на лету и безопасную эволюцию через backward compatibility (добавляю поля с default). Продюсеры и консьюмеры работают по schema_id, что экономит место в топике и гарантирует, что изменения не сломают существующих потребителей

- [ ] Что такое **consumer lag** и как с ним работать?

*Ответ одной строкой на собеседовании:* **Consumer lag**— это отставание консьюмера от продюсера (разница между последним оффсетом в партиции и закоммиченным). Мониторю через Prometheus + Burrow, алерчу при росте. Для уменьшения: оптимизирую обработку, масштабирую консьюмеров (до числа партиций), настраиваю батчинг и асинхронные коммиты. Если лаг растёт — ищу узкое место: медленная бизнес-логика, дисбаланс партиций или сетевые задержки

**Инсайт**:

  - *Лаг ≠ проблема*, если он стабильный. Небольшой постоянный лаг — норма. Опасен растущий лаг.

  - *Нельзя просто добавить консьюмеров*. Максимальный параллелизм = число партиций. Хотите масштабироваться → увеличивайте партиции (но это требует пересоздания топика или ребалансировки).

  - *Коммит — это компромисс*. Частые коммиты = меньше лаг, но больше накладных расходов. Редкие = риск потерять больше данных при краше.

- [ ] Где хранятся offsets?

*Ответ одной строкой:* Оффсеты хранятся во внутреннем топике __consumer_offsets, что даёт отказоустойчивость и изоляцию групп. Практически: консьюмеры асинхронно коммитят прогресс, поэтому при краше возможна небольшая потеря (между последним коммитом и падением), а частые коммиты создают нагрузку на системный топик

**Инсайт**:

  - *Коммит ≠ обработка.* Если вы закоммитили, но не успели обработать — при рестарте сообщение будет пропущено. Для exactly-once коммитьте после успешной бизнес-логики.

  - *enable.auto.commit=false — ваш друг*. Ручные коммиты дают контроль, но требуют аккуратности.

  - *Не игнорируйте __consumer_offsets.* При большом числе групп этот топик может разрастись — мониторьте его размер и настройте offsets.retention.minutes.

- [ ] Что такое CDC через Debezium?

**Change Data Capture (CDC)** — механизм захвата изменений данных (INSERT/UPDATE/DELETE) в источнике в реальном времени и их репликации в целевую систему.

**Debezium** — это log-based CDC: он читает транзакционные логи БД (WAL/binlog) и стримит изменения в Kafka. В отличие от polling, даёт низкую задержку, полный захват изменений (включая удаления) и минимальную нагрузку на источник. Использую с Kafka Connect + Schema Registry для надёжной доставки и эволюции схем.

**Инсайт**:

  - CDC ≠ замена бэкапам. Это механизм репликации изменений, а не полного копирования. Для первого заполнения всё ещё нужен snapshot.

  - before/after — ваша сила. Храните оба значения — это позволяет строить аудит, SCD Type 2 и откаты без усложнения бизнес-логики.

  - Мониторьте лаг коннектора. Если Debezium не успевает читать лог, источник может «раздуть» WAL/binlog и упасть из-за нехватки диска.

- [ ] Что такое **schema evolution** в потоках и почему это больно?

**Schema evolution** — процесс изменения структуры сообщений (добавление/удаление полей, смена типов, переименование) в Kafka-топике без остановки пайплайна. В стриминге данные пишутся и читаются 24/7, поэтому схема должна меняться плавно, пока старые и новые версии сервисов работают параллельно.

*Ответ одной строкой на собеседовании:* Schema evolution — это изменение структуры потоковых данных без остановки пайплайна. Больно, потому что продюсеры и консьюмеры живут в разных версиях одновременно, а поломка схемы роняет весь стриминг. Решается через Schema Registry, backward-совместимость и железное правило: только добавлять поля с default, никогда не удалять и не менять типы без плавного депрекейта.

**Инсайт для продвинутого ответа:**

  - **Схема в стриминге — это контракт на годы, а не на спринт.** Лучше добавить user_email_v2 и депрекейтнуть старое поле через полгода, чем сломать пайплайн одним ALTER.

  - **Avro/Protobuf решают 80% боли:** бинарный формат + registry + совместимость из коробки. JSON без registry в продакшене = мина замедленного действия.

  - **Мониторьте schema_registry_compatibility_errors и deserialization_failures в Prometheus.** Это первые индикаторы того, что эволюция пошла не по плану.

- [ ] Как Kafka гарантирует порядок сообщений?

Kafka гарантирует порядок только внутри одной partition по offset. Глобального порядка по всему topic нет, если partition несколько.

Поэтому для сущностей, где важен порядок, выбирают стабильный key, чтобы все события этой сущности попадали в одну partition.

- [ ] Что такое **Schema Registry** и зачем он нужен?

**Schema Registry** хранит версии схем сообщений (Avro/Protobuf/JSON Schema) и проверяет совместимость при публикации. Это снижает риск "тихих" поломок downstream при эволюции полей.

В production это обязательный слой governance для крупных event-платформ.

- [ ] Чем Debezium лучше простого polling базы?

Debezium читает журнал транзакций и получает все изменения практически в реальном времени, включая промежуточные апдейты. 

Polling обычно тяжелее для источника и может пропускать нюансы между интервалами опроса.

Но Debezium сложнее в эксплуатации, поэтому выбор зависит от требований к полноте/latency.

- [ ] Как масштабировать потребление из Kafka?

Масштабирование идет через увеличение partition и числа consumer в group. Один consumer читает одну partition в группе, поэтому максимум полезного параллелизма ограничен числом partition.

Важно балансировать количество partition заранее, чтобы не упереться в слишком дорогое пере-шардирование позже.

- [ ] Какие есть пределы масштабирования чтения у consumer group в Kafka?

Параллелизм чтения в одной группе ограничен числом partition: consumer не разделяет одну partition с другим членом группы. Увеличивать можно число partition и consumer до этого предела; дальше — только новая группа или другой топик.

Плюс брокеры и сеть — при всплеске fetch важно мониторить lag и время обработки.

- [ ] За счёт чего consumer «знает», где он остановился?

Committed offset для пары (group, topic, partition) хранится в `__consumer_offsets` (или внешнем offset store в редких настройках). Поллинг сдвигает позицию после успешной обработки (в зависимости от семантики commit).

Смена `group.id` начинает чтение с `auto.offset.reset`, что используют для replay.

## SPARK

- [ ] Что такое Apache Spark?

**Apache Spark** — распределённый движок для обработки больших данных в памяти (in-memory), поддерживающий batch, streaming, SQL, ML и Graph вычисления в едином API.

**Ключевые особенности:**

🚀 In-memory + DAG: План выполнения строится как граф, данные кэшируются в RAM → 10-100× быстрее MapReduce.

🧩 Ленивые вычисления: Трансформации (filter, join) не выполняются до action (count, write) → Catalyst Optimizer строит эффективный план.

♻️ Fault tolerance: Потерянные партиции пересчитываются по линии наследия (lineage), без репликации данных.

📦 Единый стек: DataFrame/SQL API, Structured Streaming, MLlib, GraphX — один инструмент для всех задач.

*Ответ одной строкой на собеседовании:* Spark — распределённый in-memory движок для обработки больших данных с поддержкой batch/streaming/SQL. В DE использую для ETL: читаю сырые данные из S3/Kafka, чищу и обогащаю через DataFrame API, агрегирую и пишу в витрины с партиционированием. Ключевые преимущества: ленивые вычисления + Catalyst Optimizer дают производительность, а линейность (lineage) обеспечивает отказоустойчивость без репликации

**Инсайт для продвинутого ответа:**

  - Не всё в памяти: «In-memory» ≠ «только RAM». Spark spill'ит на диск при нехватке памяти — следите за spark.memory.fraction и spark.sql.shuffle.partitions.

  - Шардинг — ключ к параллелизму: Правильный repartition() по ключу джойна/агрегации уменьшает shuffle и ускоряет запрос.

  - Structured Streaming = микро-батчи: Под капотом это тот же DataFrame API, но с checkpoint'ами и watermark'ами для exactly-once семантики.

  - Delta/Iceberg поверх Spark: Для upsert, time travel и транзакций в Data Lake использую табличные форматы, а не «голый» Parquet.

- [ ] Архитектура Spark 

Компоненты: 

**Driver** - Запускается на клиенте или в кластере. Содержит SparkSession, строит DAG, планирует задачи, агрегирует результаты. Один на приложение. Если упадёт — всё приложение падает. Память: spark.driver.memory.

**Executor** - Воркер-процесс на узле кластера. Выполняет задачи, хранит данные в кэше, отчитывается драйверу. Запускается по одному на JVM. Память делится: spark.executor.memory → Execution (shuffle/join) + Storage (cache).

**Cluster Manager** - Выделяет ресурсы под драйвер и исполнители: YARN, Kubernetes, Standalone, Mesos. Не участвует в выполнении задач, только в оркестрации ресурсов.

**SparkContext** - Низкоуровневый вход в Spark (через SparkSession). Координирует выполнение задач. Один на сессию. Через него доступны sc.parallelize(), sc.textFile().

🔄 Как выполняется запрос (жизненный цикл): 

1. Клиент отправляет код на Драйвер.

2. Драйвер (через Catalyst) строит логический план → физический план → DAG (граф зависимостей).

3. DAGScheduler разбивает DAG на Stages (границы — shuffle-операции: groupBy, join).

4. TaskScheduler отправляет Tasks (единицы работы) на Executors.

5. Executors выполняют задачи параллельно, шлют результаты/статус обратно драйверу.

6. Драйвер агрегирует финальный результат и возвращает клиенту.

**Ключевые термины:**

Job = действие (count, write), которое триггерит выполнение.

Stage = группа задач без shuffle между ними.

Task = одна единица работы над одной партицией данных.

*Ответ одной строкой на собеседовании:* Архитектура Spark — мастер-воркер: Драйвер (с SparkSession) строит DAG и планирует задачи, Cluster Manager (YARN/K8s) выделяет ресурсы, Executors параллельно выполняют задачи над партициями данных. Запрос разбивается на Jobs → Stages (по shuffle-границам) → Tasks. Память executor'а делится на Execution (для shuffle/join) и Storage (для кэша), при нехватке Execution spill'ит на диск.

- [ ] Форматы данных. RDD, DataFrame, Dataset

**RDD (Resilient Distributed Dataset)**

Низкоуровневый API (база).

  - Суть: Распределённая коллекция JVM-объектов. Самый старый интерфейс Spark.

  - Оптимизация: ❌ Нет. Spark не знает структуру данных, поэтому не может применить оптимизации (Catalyst не работает).

  - Типизация: ✅ Строгая (в Scala/Java), но проверка только в рантайме для Python.

  - Когда использовать: Только для сложной неструктурированной логики, которую невозможно выразить через DataFrame, или для работы с Legacy-кодом.

**DataFrame**

Высокоуровневый API (стандарт).

  - Суть: Распределённый набор данных с именованными колонками (схемой). Аналог таблицы в SQL или Pandas.

  - Оптимизация: ✅ Catalyst Optimizer. Spark строит логический и физический план, оптимизируя порядок операций (Predicate Pushdown, Column Pruning).

  - Движок: ✅ Tungsten. Использует off-heap память и бинарные форматы, что значительно ускоряет выполнение и уменьшает GC-паузы.

  - Когда использовать: В 95% случаев для ETL, аналитики и ML.

**Dataset**

Расширение DataFrame (Scala/Java).

  - Суть: Строго типизированная версия DataFrame. Работает с объектами JVM, но использует преимущества Catalyst и Tungsten.

  - Оптимизация: ✅ Да (как у DataFrame).

  - Типизация: ✅ Compile-time safety. Ошибки типов ловятся на этапе компиляции.

  - Нюанс для Python: В PySpark Dataset и DataFrame — это по сути одно и то же (из-за динамики Python нет строгой типизации на уровне JVM). Различие актуально только для Scala/Java.

*Ответ одной строкой на собеседовании:* RDD — это низкоуровневая коллекция объектов без оптимизации, используется редко. DataFrame — это структурированные данные со схемой, которые оптимизируются движком Catalyst и выполняются на Tungsten, что даёт прирост скорости в разы. Dataset — это типобезопасное расширение DataFrame для Scala/Java. В PySpark я использую почти всегда DataFrame, так как он сочетает удобство SQL и высокую производительность

**Инсайт для продвинутого ответа:**

  - *Почему DataFrame быстрее RDD?* Потому что Spark видит схему. Он может отбросить ненужные колонки (Column Pruning) и протолкнуть фильтры к источнику (Predicate Pushdown). В RDD Spark читает всё и фильтрует в памяти.

  - Encoder: В Dataset данные хранятся в компактном бинарном формате (не Java-сериализация), что экономит память и CPU.

  - Эволюция: Раньше писали на RDD → сейчас пишем на DataFrame/Dataset → будущее за Spark Connect (клиент-серверная архитектура).

- [ ] Чем отличается RDD от DataFrame?

**3 главных отличия (что ждут услышать)**

**Оптимизация (Catalyst)**

DataFrame знает схему → Spark строит логический план, применяет правила оптимизации (Predicate Pushdown, Column Pruning, Join Reordering) и генерирует эффективный физический план. RDD таких оптимизаций не имеет.

**Память и сериализация (Tungsten)**

RDD хранит обычные объекты JVM/Python → тяжёлая сериализация, частые паузы GC. DataFrame хранит данные в компактном бинарном формате вне кучи (off-heap) → меньше нагрузка на GC, быстрее чтение/запись.

**Схема данных**

DataFrame = Row + Schema. Это позволяет Spark валидировать запросы до запуска, автоматически определять типы и оптимизировать хранение. В RDD схема неизвестна, всё проверяется в рантайме.

*Ответ одной строкой на собеседовании:* RDD — низкоуровневая распределённая коллекция объектов без встроенной оптимизации. DataFrame — структурированные данные со схемой, которые оптимизируются движком Catalyst и выполняются на Tungsten, что даёт прирост скорости в разы, экономию памяти и удобный SQL-подобный API. В современных пайплайнах использую почти всегда DataFrame.

- [ ] Что такое SparkSession?

**SparkSession** — единая точка входа в приложение Apache Spark (начиная с v2.0). Объект, который создаётся на драйвере и даёт доступ ко всем возможностям фреймворка: DataFrame API, SQL, Streaming, Catalog, конфигурации.

*Ответ одной строкой на собеседовании:* SparkSession — это единая точка входа в Spark 2+, объединяющая SparkContext, SQLContext и HiveContext. Через неё создаются DataFrame, выполняются SQL-запросы, управляется каталог и настройки приложения. В коде инициализируется один раз на драйвере через builder.getOrCreate() и в конце явно закрывается stop() для освобождения ресурсов кластера

- [ ] Какие основные JOIN в Spark (DataFrame/SQL) чаще всего ждут на собеседовании?

**Физические стратегии (Execution Plans):**

Это то, как Spark реально выполняет JOIN. Оптимизатор выбирает стратегию автоматически.

**BroadcastHashJoin (BHJ)** - Если одна таблица маленькая (обычно < 10 МБ). Маленькая таблица рассылается на все узлы (Broadcast), большая читается локально.

**SortMergeJoin (SMJ)** - Если обе таблицы большие. Дефолт. Данные сортируются по ключу JOIN и перемешиваются (Shuffle).

**ShuffleHashJoin (SHJ)** - Если одна таблица не влезает в Broadcast, но меньше другой. Строится хеш-таблица для одной стороны, другая сторона прогоняется через неё.

**CartesianProduct** - Если нет условия ON (или cross join). Каждый элемент с каждым.

*Специальные JOIN-типы в Spark*:

В Spark DataFrame API есть типы, которых нет в классическом SQL или которые пишутся короче.

  - **left_semi**: Возвращает строки из левой таблицы, для которых нашлось совпадение в правой (аналог WHERE EXISTS). Не дублирует строки, если в правой таблице несколько совпадений.

  - **left_anti**: Возвращает строки из левой таблицы, для которых нет совпадений в правой (аналог WHERE NOT EXISTS или NOT IN).

  - **cross**: Декартово произведение.

*Ответ одной строкой на собеседовании:* В Spark есть логические типы (Inner, Left, Semi) и физические стратегии. Для маленьких таблиц используется быстрый BroadcastHashJoin (без shuffle), для больших — SortMergeJoin (по умолчанию). Главная боль — Data Skew, лечится солью (salting) или AQE. Также активно использую left_anti для дедупликации вместо NOT IN.

- [ ] Как ускорить JOIN большой таблицы с маленькой справочной?

Нужно форсировать **BroadcastHashJoin**.

1. Настроить порог: spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "10485760") (10 МБ).
2. Явно указать хинт: df_large.join(broadcast(df_small), "key").

- [ ] Какие основные трансформации в Spark (DataFrame/SQL) чаще всего ждут на собеседовании?

В Spark трансформации делятся на **узкие (narrow)** и **широкие (wide)**. На собеседовании ждут не просто перечисления методов, а понимания их влияния на shuffle, границы Stage и оптимизации Catalyst.

**Узкие трансформации (No Shuffle):**

Выполняются в пределах одной партиции. Дешёвые, не запускают новые Stage.

select() - Выбор/переименование колонок

filter() / where() - Фильтрация строк

withColumn() - Добавление/изменение колонки

drop(), cast(), na.fill() - Удаление, приведение типов, заполнение NULL

**Широкие трансформации (Shuffle):**

Требуют перемешивания данных между executor'ами. Определяют границы Stage, самые дорогие.

groupBy().agg() - Агрегация по ключу

join() - Соединение таблиц

orderBy() / sort() - Глобальная сортировка

distinct() / dropDuplicates() - Удаление дублей

repartition() / coalesce() - Изменение числа партиций

**Структурные / Сложные**

explode() - Разворачивание массивов/JSON

pivot() / unpivot() - Строки ↔ столбцы

union() vs unionByName() - Объединение датафреймов

Оконные функции - row_number(), lag(), скользящие суммы

*Ответ одной строкой на собеседовании:* Трансформации делятся на узкие (без shuffle: select, filter, withColumn) и широкие (с shuffle: groupBy, join, orderBy, distinct). Spark выполняет их лениво, строя DAG, а Catalyst автоматически применяет predicate pushdown и column pruning.

- [ ] Почему withColumn с Python-функцией тормозит?

Каждая строка сериализуется из JVM в Python-процесс и обратно. Решение: встроенные функции Spark или pandas_udf (Arrow).

- [ ] Чем distinct() отличается от dropDuplicates()??

distinct() хеширует все колонки, dropDuplicates(cols) — только указанные. Второе быстрее и потребляет меньше памяти.

- [ ] Зачем делать repartition перед write??

Чтобы избежать проблемы мелких файлов. Если задач 1000, а партиций в write 1000 → 1000 файлов по 1 МБ. repartition(50) даст 50 файлов по ~200 МБ → эффективнее для Parquet/ClickHouse.

- [ ] Почему Spark пришел на замену классическому MapReduce?

**Как работает MapReduce (3 шага):**

1. Map — каждая задача обрабатывает входные данные, генерирует пары key → value.
2. Shuffle & Sort — данные группируются по ключу, передаются по сети между узлами.
3. Reduce — агрегация сгруппированных данных в финальный результат.

*Ключевая особенность:* После каждой фазы промежуточные результаты записываются на диск (HDFS). Это обеспечивает отказоустойчивость, но создаёт bottleneck.

Spark старается переиспользовать данные в памяти (с spill на диск при нехватке RAM), имеет DAG scheduler и высокоуровневые API (SQL, DataFrame, Structured Streaming), что ускоряет типичные аналитические и ML-циклы и снижает инженерный overhead.

*Ответ одной строкой на собеседовании:* MapReduce — двухфазная парадигма с записью промежуточных результатов на диск, что делает её медленной для сложных пайплайнов. Spark заменил её благодаря in-memory вычислениям, DAG-движку и единому API, давая прирост скорости в разы и поддержку стриминга/ML без перезапуска джобов

- [ ] Что такое **lazy evaluation** в Spark?

Ленивые вычисления (lazy evaluation) означают: трансформации (`select`, `filter`, `join`…) не запускают работу кластера сразу — они лишь наращивают граф зависимостей (lineage/DAG). Реальное выполнение начинается на action: `count`, `collect`, `write`, `show`, `take` и т.д.

Плюс: Catalyst видит весь план и может свернуть шаги, протолкнуть фильтры, обрезать столбцы (column pruning). Минус для новичков: исключение по плохим данным может всплыть только на action, а не в момент объявления `withColumn`.

- [ ] Что такое **shuffle** и почему он дорогой?

**Shuffle** — перераспределение данных между executor’ами при groupBy, join, repartition, distinct, orderBy. Самая дорогая операция в Spark (запись на диск → передача по сети → чтение).

*Ответ одной строкой на собеседовании:* Shuffle — самая ресурсоёмкая операция в Spark. Главные проблемы: data skew (один таск висит), OOM/spill (нехватка памяти), disk/network bottlenecks и overhead от мелких партиций. Диагностирую через Spark UI (Exchange metrics, spill, task duration), решаю через AQE, salting, настройку shuffle.partitions, broadcast для малых таблиц и сжатие

- [ ] Что такое **spill** в Spark?

Spill — вынужденная выгрузка промежуточных структур (shuffle sort, hash table aggregation, sort merge join) на диск, когда не хватает heap/`off-heap` в `execution`-зоне. Диск на порядки медленнее RAM ⇒ stage резко растёт по времени.

Что делать инженеру: уменьшить размер строк и объём данных до тяжёлого `join` (`filter`/проекция), исправить skew, подобрать `spark.sql.shuffle.partitions`, включить AQE, не злоупотреблять cartesian-паттернами, при необходимости дать ресурсы executor-ам.

- [ ] Что такое **data skew** и как с ним бороться?

Skew (перекос) — когда один (или немного) ключ `join`/`groupBy` содержит гигантскую долю строк; соответствующая task выполняется часами, остальные простаивают — стадия упирается в straggler.

Типовые приёмы: salting (искусственно размазать горячий ключ по суффиксам с последующей догонной агрегацией), двухшаговая агрегация (pre-aggregate «холодных» ключей отдельно), AQE skew join (если доступно и настроено), смена модели (другой ключ, broadcast для «большой» редкой справки осторожно), фильтрация выбросов по доменным правилам.

- [ ] **repartition** vs **coalesce**?

**repartition(n)** почти всегда делает shuffle и переразбивает данные на n партиций более равномерно — хорошо перед тяжёлым `join`/`groupBy`, если исходные партиции кривые. 

**coalesce(n)** сжимает число партиций без полного shuffle (склеивает существующие), что дешевле, но может оставить «толстые» партиции и хуже параллелизм.

Паттерн: `repartition` «выровнять» перед compute, `coalesce` перед `write`, чтобы не плодить мелкие файлы в lake.

- [ ] Когда и зачем использовать `cache`/`persist`?

Если один и тот же промежуточный DataFrame читается в нескольких ветках плана (несколько downstream действий), lineage по умолчанию пересчитает его заново. `cache`/`persist` материализуют storage-уровень (память/диск/сериализованный формат) и сокращают повторную работу.

Важно: `unpersist` после использования — иначе storage конкурирует с execution и может вытеснять полезный кэш. Не кэшируйте гигантские датасеты «на всякий случай».

- [ ] Что такое **broadcast join**?

Broadcast hash join — маленькая таблица целиком посылается на все executor-ы (через TorrentBroadcast с driver), и большая сторона локально пробивается по hash без тяжёлого shuffle большой таблицы.

Риски: размер broadcast ограничен `spark.sql.autoBroadcastJoinThreshold` и памятью; ошибочно выбранный broadcast убьёт executor OOM. Следите за статистикой и фактическим планом (`EXPLAIN`/UI).

- [ ] Почему **Python UDF** (User-Defined Function - Пользовательская функция) — часто плохая идея?

Это кастомная функция, которую пишет разработчик, чтобы расширить возможности SQL или распределённого движка (Spark, Hive, Presto, ClickHouse), когда встроенных операторов (sum, concat, date_trunc) не хватает для реализации бизнес-логики.

**Стандартный UDF (Row-by-Row)** - Используется для простой логики, но медленный из-за сериализации между JVM и Python.

**Pandas UDF (Vectorized) — "Золотой стандарт"** - Используется для тяжелой логики или больших данных. Работает в 10–100 раз быстрее за счет векторизации (Apache Arrow).

Стандартный UDF вызывает Python Worker для обработки каждой строки. Данные сериализуются из JVM (Java) в Python, обрабатываются, и результат возвращается обратно. Это создает огромный оверхед.
Решение: Использовать Pandas UDF (Vectorized UDF), который передает данные блоками (батчами) через Apache Arrow, минимизируя сериализацию и используя нативную производительность Pandas/NumPy

- [ ] Что такое AQE (Adaptive Query Execution)?

**AQE** — механизм в Apache Spark 3.0+, который динамически оптимизирует план запроса во время выполнения на основе реальных статистик, собранных после завершения предыдущих стадий (stages). Вместо слепой reliance (estimates) до запуска, Spark «учится» на фактических данных и перестраивает оставшуюся часть пайплайна.

**3 ключевые возможности:**

*Dynamic Partition Coalescing* - Объединяет мелкие shuffle-партиции в более крупные после завершения shuffle (Убирает проблему «миллиона мелких задач», снижает overhead на TaskScheduler и проблему мелких файлов при записи)

*Dynamic Join Strategy Switching* - Меняет стратегию JOIN на лету (например, SortMergeJoin → BroadcastHashJoin). Если после фильтрации/агрегации таблица стала маленькой, AQE автоматически выберет broadcast, экономя shuffle

*Skew Join Optimization* - Разбивает skewed-партиции на несколько мелких подзадач. Устраняет «straggler»-эффект без ручного salting или фильтрации NULL

🔄 **Как работает (механика):**
1. Spark запускает первые стадии плана (обычно до первого shuffle).
2. После завершения стадии собирает runtime-метрики: реальный размер партиций, количество строк, распределение ключей.
3. Оптимизатор перестраивает план для следующих стадий с учётом этих метрик.
4. Процесс повторяется после каждого shuffle-границы.

*Ответ одной строкой на собеседовании:* AQE (Adaptive Query Execution) — runtime-оптимизатор Spark 3.0+, который собирает реальные статистики после shuffle и динамически коалесцирует партиции, переключает стратегии JOIN и разбивает skewed-данные без ручного тюнинга. Включается одной настройкой и в современных версиях работает по умолчанию

- [ ] Что критично для Structured Streaming?

**Structured Streaming** — потоковая часть Spark, завязанная на DataFrame API и checkpoint в надёжном хранилище (HDFS/S3/…). Три опоры, без которых «не поедет» в проде:

1. Checkpoint — сохраняет offset источника (например, Kafka), состояние stateful-операций (агрегации по окну), метаданные write-ahead к sink, чтобы после failover продолжить идемпотентно и не дублировать произвольно.

2. Watermark (watermark on event time) — граница опоздания событий: состояние старее воды может выбрасываться, чтобы state не рос бесконечно. Это прямой компромисс полнота vs ресурсы/латентность.

3. Output mode (`append`, `complete`, `update`) — что именно пишем в sink на каждом trigger micro-batch и как это сочетается с идемпотентностью приемника.

Без явной политики на late events отчёты по окнам «плавают» и не согласуются с бизнесом.

- [ ] **Параллелизм** в Spark

Параллелизм (Parallelism) в Spark = количество задач (tasks), на которые разбивается работа. Одна задача обрабатывает ровно одну партицию данных.

*Ответ одной строкой на собеседовании:* Параллелизм в Spark определяется числом партиций: одна партиция = одна задача. Минимально возможный — 1, но это антипаттерн. По дефолту для DataFrame используется 200 партиций (spark.sql.shuffle.partitions), для RDD — spark.default.parallelism. Важно отличать логический параллелизм (общее число задач) от физического (сколько ядер доступно одновременно). На практике подбираю ~128 МБ–1 ГБ данных на партицию, чтобы избежать оверхеда от мелких задач или bottleneck'а от слишком малого числа.

- [ ] Какие виды **кэширования** существуют в Spark и чем они отличаются?

Механизм сохранения промежуточных результатов вычислений в памяти/на диске executor'ов, чтобы не пересчитывать их при повторных action-ах.

В Spark есть cache() (шорткат для MEMORY_ONLY) и persist(level) с выбором уровня: RAM, Disk, сериализация. cache() работает лениво — данные сохраняются только после первого action. В продакшене чаще использую MEMORY_AND_DISK или MEMORY_ONLY_SER для баланса памяти/CPU, и всегда вызываю unpersist() после использования.»

- [ ] Чем **map** отличается от **flatMap**?

На RDD/Dataset: `map` применяет функцию 1→1 к каждому элементу. `flatMap` — 1→много: функция возвращает итерируемое (0, 1 или N элементов), Spark склеивает в плоский поток. Для понимания кардинальности пайплайна это база: `flatMap` может расширить или сжечь поток строк.

На DataFrame чаще используют SQL/встроенные функции и `explode` вместо ручного flatMap.

- [ ] Как устроено управление памятью в Spark (storage vs execution)?

«В Spark память executor'а делится на Execution (временные вычисления: shuffle, join, sort) и Storage (кэш, broadcast). Они динамически заимствуют свободную память друг у друга, но Execution имеет приоритет: при нехватке вытесняет кэш на диск. Настраиваю через spark.memory.fraction (0.6) и storageFraction (0.5), а переполнение Execution лечу увеличением executor memory или оптимизацией skew, чтобы избежать spill.»

- [ ] Чем **cache()** отличается от **persist()**?

`cache()` — синоним `persist(MEMORY_AND_DISK_DESER)` или `MEMORY_ONLY` в зависимости от версии/конфига документации (уступим практическому правилу: «дефолтный короткий путь без явного выбора уровня»). `persist(StorageLevel)` позволяет явно указать `MEMORY_ONLY`, `MEMORY_AND_DISK`, `DISK_ONLY`, serialized‑уровни — что важнее при тяжёлом объёме и давлении на heap.

Выбирайте `persist`, когда важнее контролировать eviction и размер объектов (serialized может съесть CPU, но сэкономить RAM).

- [ ] Что такое Catalyst Optimizer и Tungsten Engine?

**Catalyst Optimizer (Планировщик)** - Интеллектуальный фреймворк оптимизации запросов для DataFrame/SQL API.
Преобразует ваш код в план выполнения и улучшает его до запуска:

1. Unresolved Logical Plan → привязка к схеме/каталогу
2. Analyzed Logical Plan → проверка типов, разрешённые колонки
3. Optimized Logical Plan → применение правил (Rule-Based + Cost-Based)
4. Physical Plan(s) → выбор стратегий (BroadcastJoin vs SortMerge и т.д.)
5. Compiled Bytecode → генерация Java-кода для выполнения

**Tungsten Engine (Исполнитель)** - Низкоуровневый движок выполнения, оптимизированный под современное железо (CPU + память).

*Ответ одной строкой на собеседовании:* Catalyst — оптимизатор запросов, который до запуска строит эффективный план (pushdown, pruning, выбор join-стратегии). Tungsten — низкоуровневый движок выполнения, который запускает этот план с учётом CPU-кэшей, off-heap памяти и генерацией единого байткода. Вместе они делают DataFrame в разы быстрее RDD, снимая с разработчика рутину по ручной оптимизации

- [ ] Что такое micro-batch processing в Structured Streaming?

Micro-batch — поток обрабатывается пакетами по триггеру (`ProcessingTime` каждые N секунд, `Once`, `Continuous` — отдельный режим). Внутри батча reuse-ится почти тот же движок, что и в batch Spark SQL, плюс checkpoint/state store.

Для DE это обычно минутные SLA, а не субсекундный CEP; зато проще операционка и exactly-once к idempotent sink согласуемее, чем у чистого record-by-record стрима.

- [ ] Как обрабатывать late data в стриминге?

Late data — событие пришло позже, чем вы уже закрыли агрегат по event time с учётом watermark. Стратегии: ограничить опоздание watermark’ом (события слишком старые drop), сложить в DLQ/отдельный топик для ручного/batch пересчёта, либо вести двухконтурную систему: быстрый неточный слой + ночной корректирующий batch по сырью. Sink должен быть идемпотентным (merge по ключу).

С бизнесом фиксируют: какая максимальная задержка допустима для оперативной метрики и что делаем с «навсегда опоздавшими» событиями.

- [ ] Что такое SparkContext и зачем он в приложении?

`SparkContext` — «корневой» объект Spark: подключение к cluster manager, SparkConf, создание RDD, аккумуляторы, broadcast variables, task scheduling на уровне application. В Spark 2+ чаще создают `SparkSession`, который оборачивает `SparkContext` для SQL/DataFrame.

Формула для ответа: `SparkSession` — удобный фасад для DE; `SparkContext` — низкоуровневый жизненный цикл приложения.

- [ ] Чем DataFrame API отличается от RDD?

RDD (Resilient Distributed Dataset) — низкоуровневые распределённые коллекции с ручным `partition`, coarse-grained transformations; оптимизатора SQL нет, schema нет. DataFrame — табличное представление со схемой; Dataset[T] (в Scala/Java) добавляет типобезопасность. Оба DataFrame/Dataset проходят Catalyst/Tungsten.

Для ETL предпочитают SQL/DataFrame; RDD — нестандартные операции, legacy, интеграции, где нужен низкий контроль.

- [ ] Где на практике применяют Spark Streaming / Structured Streaming?

Типичные кейсы: агрегации Kafka-потоков в витрины, обогащение событий справочниками (broadcast осторожно), контроль SLA доставки, микробатч-запись в Delta/Iceberg/Hudi или OLAP через staging таблицы, аномалии и алерты по реальному времени в мягком смысле.

В проде обязательны: checkpoint, watermark, идемпотентная запись, мониторинг lag и state size, тест failover.

- [ ] Почему сначала `filter`, потом агрегация, а не наоборот?

Ранний `filter` уменьшает число строк до shuffle и агрегатных буферов ⇒ меньше сеть/CPU/RAM. Поздний фильтр после глобальной агрегации может заставить вас считать всю историю, если оптимизатор не смог продвинуть предикат (не всегда возможно из‑за семантики выражения).

Правило: пишите запросы так, будто оптимизатора нет, и проверяйте `EXPLAIN`.

- [ ] Как отличить transformation от action в Spark на примере кода?

Transformation строит DAG и не запускает jobs (пока не потребуется output): `select`, `filter`, `join`, `groupBy`, `withColumn`, `repartition`… Action триггерит выполнение: `count`, `take`, `collect`, `foreach`, `write`, `save`…

Если в конце скрипта нет action, кластер ничего не посчитал — частая ошибка новичков при отладке в REPL.

- [ ] Что такое бакетирование (bucketing) данных в Spark и зачем оно?

Таблица сохраняется с фиксированным числом файлов/партиций по `hash(bucket_cols) % numBuckets`, иногда с сортировкой внутри бакета. Повторные join/агрегации по тому же ключу могут избежать полного shuffle, если обе стороны согласованно забакетированы.

Это trade-off: нужно поддерживать согласованность numBuckets и колонок; при смене данных пересчёт бакетов может быть дорогим.

- [ ] Как распараллелить чтение из PostgreSQL/Greenplum в Spark по JDBC?

Используют `partitionColumn`, `lowerBound`, `upperBound`, `numPartitions`: Spark создаёт диапазоны по колонке и открывает несколько JDBC-коннектов параллельно. Это снимает узкое место «один поток читает всю таблицу».

Нужен числовой или равномерный ключ (часто surrogate id). Слишком много партиций перегружает источник, поэтому число коннектов согласуют с DBA и лимитами БД.

- [ ] Как устроена иерархия Job, Stage и Task в Spark?

Один action запускает job. Job делится на stages по границам shuffle (wide dependencies). Task — минимальная единица на одной partition внутри stage.

В UI по длительности task видно skew: одна task тянет весь stage.

- [ ] Сколько executor-процессов разумно на одной ноде и от чего зависит параллелизм?

Часто 1–2 executor на машину, чтобы не дробить память и диск между множеством JVM и не ловить contention. Параллелизм задач в момент времени ограничен `executor_instances × executor_cores` и числом партиций.

Тюнинг — баланс между крупными executors (shuffle heavy) и большим числом мелких.

- [ ] В чём разница `client mode` и `cluster mode` в Spark-submit?

В `client mode` driver работает на машине, откуда запустили приложение; логи и жизненный цикл зависят от этой машины и сети до кластера. В `cluster mode` driver поднимается внутри кластера, что устойчивее для production и долгих batch-задач.

Для интервью: `client` удобен для разработки/отладки, `cluster` обычно выбирают для надёжного прод-запуска.

- [ ] Чем плох shuffle в Spark «в одном абзаце»?

Shuffle — это сериализация, запись промежуточных данных, сеть между узлами и merge на стороне reader; он задаёт границы stages и часто доминирует во времени и в чувствительности к skew.

Поэтому broadcast, ранние фильтры, AQE и правильный ключ партиционирования — главные рычаги.

- [ ] Чем Spark принципиально отличается от Greenplum/MPP-СУБД?

Spark — вычислительный движок поверх файлов/источников без встроенного долговечного хранилища и без полноценных OLTP-транзакций на уровне таблиц в кластере. MPP-СУБД хранит данные, оптимизирует SQL, индексы и конкурентный доступ аналитиков.

Типичный тандем: Spark для тяжёлой трансформации в lake, Greenplum/ClickHouse для сервинга витрин и BI.

## Hadoop/HDFS

- [ ] Что входит в Hadoop-экосистему на базовом уровне?

Ядро — HDFS для хранения и YARN для управления ресурсами. Поверх обычно работают Hive/Spark и другие сервисы обработки и каталогизации.

В собеседовании важно показать, что ты понимаешь роли, а не просто перечисляешь названия компонентов.

- [ ] Зачем в экосистеме Hadoop отдельно выделяют Hive и Metastore?

Hive даёт SQL-слой и табличную абстракцию над файлами в HDFS (внешние/управляемые таблицы), а Metastore хранит каталог: схемы, партиции, местоположение файлов. Spark и другие движки часто читают те же метаданные через совместимый каталог.

Без понятия metastore сложно объяснить, как партиции попадают в план запроса и почему «просто положили parquet» ещё не означает управляемую таблицу.

- [ ] Как работает HDFS при чтении файла?

Клиент обращается к NameNode за метаданными (где лежат блоки), после этого читает сами блоки с DataNode. Данные читаются по репликам, обычно выбирается ближайшая.

NameNode не хранит пользовательские данные, только метаданные. Это критично для понимания узких мест HDFS.

- [ ] Зачем нужен фактор репликации?

Он определяет число копий блока на разных узлах и отвечает за отказоустойчивость. Стандартно ставят 3 как компромисс между надежностью и стоимостью хранения.

При сбое узла кластер восстанавливает требуемое число реплик, чтобы соблюсти policy.

- [ ] Что такое проблема мелких файлов в HDFS?

Даже маленький файл создает metadata overhead. Если файлов миллионы, NameNode и метастор деградируют, и чтение становится медленным.

Поэтому на практике данные периодически компактуют в более крупные файлы, особенно в холодных партициях.

## GREENPLUM

- [ ] Что такое MPP-архитектура и зачем она нужна?

MPP делит данные и вычисления между множеством узлов по shared-nothing модели. Это позволяет обрабатывать большие объемы аналитики параллельно.

Цена за масштаб — сложность распределения данных и сетевого обмена. Неправильный design может «съесть» выигрыш от параллелизма.

- [ ] Архитектура GreenPlum

Ключевые компоненты: 

**Master** - Принимает запросы, строит план, распределяет работу, агрегирует результаты. Не хранит пользовательские данные.

**Segment** - Рабочая лошадка: хранит часть данных, выполняет локальные операции (scan, filter, join, agg). Запускается как отдельный PostgreSQL-процесс.

**Interconnect** - Сетевой слой для обмена данными между сегментами (shuffle при JOIN / GROUP BY).

**Mirror Segment** - Реплика сегмента на другом хосте для отказоустойчивости (автоматический failover).

Как выполняется запрос: 

1. Parse & Plan (Master): Оптимизатор строит параллельный план.

2. Dispatch: Master отправляет подзапросы на все сегменты.

3. Local Execution (Segments): Каждый сегмент обрабатывает свою часть данных.

4. Motion (Interconnect): При необходимости данные пересылаются между сегментами (например, для глобальной агрегации).

6. Finalize (Master): Сборка финального результата и отправка клиенту.

- [ ] Что такое **distribution key** в Greenplum и как его выбирать?

При создании таблицы указывается DISTRIBUTED BY (key):

**Distribution key** определяет, на какой сегмент попадет строка. В идеале выбирают ключ, который часто используется в join, чтобы избежать лишнего redistribute motion.

Если ключ выбран плохо, возникает skew или дорогие сетевые пересылки. Поэтому выбор ключа — архитектурное решение, а не мелкий тюнинг.

- [ ] Типы Motion-операций в Greenplum?

**Gather** — собрать результаты со всех сегментов на мастер (для LIMIT, ORDER BY).

**Broadcast** — отправить маленькую таблицу на все сегменты (для JOIN).

**Redistribute** — перераспределить данные по новому ключу (для JOIN по другому полю).


- [ ] Что такое data skew в Greenplum?

Это неравномерность данных по сегментам, когда один сегмент содержит значительно больше строк/работы, чем остальные. Итоговый runtime упирается в самый медленный сегмент.

Диагностика обычно делается через `gp_segment_id` и сравнение распределения строк. Лечение — пересмотр distribution key или стратегии загрузки.

- [ ] Когда в Greenplum выбирают `DISTRIBUTED RANDOMLY` и в чём цена?

Случайное распределение выравнивает объём по сегментам, если подходящего высококардинального ключа нет или он сильно перекошен. Цена — почти любой `JOIN` по бизнес-ключу потребует redistribute motion: данные не co-located, сеть становится узким местом.

Используют как компромисс для «сырых» staging-таблиц или когда важнее равномерная загрузка, чем локальные join.

- [ ] Когда используют `DISTRIBUTED REPLICATED` в Greenplum?

Для маленьких справочников, которые часто участвуют в join с большими фактами. Копия таблицы хранится на каждом сегменте, и это позволяет выполнять join локально без broadcast/redistribute этой маленькой таблицы на каждом запросе.

Цена — рост объёма хранения и нагрузки на обновление справочника, поэтому replicated подходит только для действительно небольших и редко меняющихся таблиц.

- [ ] Чем partitioning отличается от distribution в Greenplum?

Distribution отвечает за раскладку строк между сегментами кластера, а partitioning — за логическое деление таблицы (обычно по времени) внутри хранения для pruning и retention.

Эти механизмы независимы и используются вместе: например, distributed by customer_id и partition by month.

- [ ] Что такое motion в плане Greenplum?

Motion — сетевой обмен данными между сегментами. Основные типы: redistribute, broadcast, gather. Это самые «дорогие» операции в распределенных планах.

Поэтому при оптимизации в GP часто задача №1 — сократить motion и держать joins локальными там, где возможно.

- [ ] Какие типы таблиц хранения часто используют в Greenplum?

Для аналитики часто используют append-optimized таблицы (row/column), для некоторых сценариев — heap. Выбор зависит от паттерна чтения/записи и компрессии.

На DWH-нагрузке column-oriented storage обычно дает лучший компромисс по скорости чтения и объему хранения.

- [ ] Что важно помнить про Append-Optimized (AO) таблицы в Greenplum?

AO-таблицы хороши для append-heavy аналитики: лучше компрессия и быстрые вставки больших батчей. Но частые `UPDATE/DELETE` для них дороже, потому что паттерн ближе к переписыванию/очистке и последующему vacuum.

Практический вывод: AO выбирают для фактов и исторических слоёв, а write-heavy сценарии с частыми точечными изменениями обычно оставляют на heap/другой дизайн пайплайна.

- [ ] Какие основные способы загрузки данных в Greenplum?

Базовые варианты: `COPY` (из файлов/STDIN), external tables через `gpfdist`, интеграция через PXF (HDFS/S3/JDBC и др.), а также ELT-подход через staging-таблицы и дальнейший `INSERT ... SELECT`/`MERGE`-паттерны. Выбор зависит от объема, SLA и источника.

На больших объемах обычно строят двухшаговую схему: быстрая загрузка в staging с минимальными проверками, затем контролируемое преобразование в целевые таблицы с quality checks и статистикой.

- [ ] Как обычно оптимизируют тяжелый SQL в Greenplum?

Оптимизация в Greenplum строится вокруг MPP-архитектуры: если запрос тормозит, в 90% случаев проблема в неравномерной нагрузке на сегменты, лишних пересылках данных или устаревшей статистике.

Смотрят `EXPLAIN ANALYZE`, проверяют motion/nested loop/skew, разбивают монолитный запрос на этапы с промежуточными таблицами, выбирают только нужные колонки, обновляют статистику.

Сильный ответ — это последовательность диагностики, а не список «магических» параметров.

*Ответ одной строкой на собеседовании:* Начинаю с EXPLAIN ANALYZE: убираю skew через правильный DISTRIBUTED BY, минимизирую Redistribute/Broadcast motion, перевожу таблицы в AO/CO со сжатием ZSTD, включаю partition pruning, обновляю статистику через ANALYZE и настраиваю Resource Groups + statement_mem под workload. Индексы в GP для аналитики не использую

**Инсайты для продвинутого ответа:**
  - Skew убивает MPP: если 1 сегмент работает 10 минут, а 15 простаивают → запрос будет идти 10 минут. Распределение важнее индексов.
  - ANALYZE > индексы: в колоночных AO-таблицах статистика min/max + histogram дают Predicate Pushdown и отсекают целые блоки. Без ANALYZE оптимизатор слеп.
  - Избегайте UPDATE/DELETE в AO: они создают hidden rows и раздувают таблицу. Лучше паттерн INSERT new → TRUNCATE old → RENAME или использование gp_inject_fault только в тестах.
  - Workfile limit: если statement_mem не хватает, запрос падает с could not write to workfile. Настраивайте gp_vmem_protect_limit и statement_mem под пиковые агрегации.
  - Greenplum 7: перешёл на PostgreSQL 12+, улучшил оптимизатор, добавил Resource Groups вместо legacy Queues, ускорил AO-таблицы. Упоминайте это как плюс к актуальности знаний.

- [ ] Что такое spill в Greenplum и как его снижать?

Spill — это выгрузка промежуточных данных join/sort/aggregate на диск из-за нехватки памяти, что резко увеличивает latency. Обычно проявляется на больших группировках и сложных join-планах.

Снижают spill ранней фильтрацией, сокращением ширины строк, разбиением запроса на этапы, корректным распределением данных (меньше motion) и настройкой memory-параметров в рамках политики кластера.

- [ ] Почему запрос с большим числом CTE в Greenplum иногда «тонет» в spill и как это лечить?

Цепочка крупных CTE может материализовать много промежуточных наборов; при недостатке памяти каждый этап уходит в disk spill, а затем ещё и гоняет данные через motion между сегментами. В результате запрос растёт нелинейно по времени.

Практика: как можно раньше фильтровать и сужать колонки, тяжёлые этапы выносить во временные таблицы со статистикой, а не держать всё в одном монолите CTE.

- [ ] Зачем в Greenplum зеркала (mirrors) сегментов?

Mirror — резервная копия primary-сегмента на другом хосте: при падении узла кластер переключается на зеркало и сохраняет доступность данных. Это про HA, не про ускорение чтения.

В мини-кластере из двух нод зеркала планируют так, чтобы primary и mirror одного контента не сидели на одном физическом сервере.

- [ ] Для чего нужен PXF в Greenplum?

PXF (Platform Extension Framework) — встроенный мост для external tables: читать/писать HDFS, S3, JDBC и другие источники через единый SQL-слой, не копируя всё предварительно локально на сегменты.

В DE это типичный путь ingest из lake или внешних БД с pushdown где возможно.

- [ ] Как на интервью обычно формулируют CAP для OLAP (ClickHouse vs Greenplum)?

CAP — не «выбери два навсегда», а компромиссы под нагрузку. ClickHouse и подобные колоночные движки часто ослабляют межзапросную строгую консистентность (асинхронные merge/replica lag) ради latency и throughput. Greenplum остаётся классической MPP-СУБД с транзакциями на уровне кластера, но распределённые блокировки и двухфазный commit всё равно дороги.

Важно говорить про ожидания SLA по свежести данных и изоляции, а не цитировать CAP как догму.

## ClickHouse

- [ ] Почему ClickHouse быстрый на аналитике?

Он хранит данные колонками, хорошо сжимает и выполняет операции векторно. Это эффективно для агрегаций и чтения небольшого набора колонок из больших таблиц.

Но скорость сильно зависит от правильного моделирования таблиц, а не только от «сам по себе быстрый движок».

- [ ] Что такое `ORDER BY` в таблице MergeTree и почему это важно?

`ORDER BY` определяет физический порядок данных и ключ sparse-индекса. От него зависит, насколько эффективно будут отсеиваться диапазоны при фильтрации.

Неподходящий ключ ведет к большим сканам и деградации, даже если железо мощное.

- [ ] Что такое кардинальность колонки в ClickHouse и как она влияет на модель таблицы?

Кардинальность — число уникальных значений. Очень низкая (пол, boolean) в начале `ORDER BY` почти не помогает отсечению гранул. Очень высокая уникальность (UUID в начале ключа) может размазать данные по частям и ухудшить сжатие и merge без выигрыша в типичных фильтрах.

Практика: первые колонки ключа — те, по которым почти всегда фильтруют и которые имеют «умеренную» селективность; высококардинальные поля иногда выносят в secondary indexes или в конец ключа.

- [ ] Что такое part и merge в ClickHouse?

Каждая вставка формирует part — физический immutable кусок данных. Фоновые merge-процессы объединяют parts и выполняют сервисную работу (в том числе в ReplacingMergeTree).

Если вставки слишком мелкие и частые, parts становятся слишком многочисленными, и система начинает тратить ресурсы на постоянные merge.

- [ ] Какие основные способы загрузки данных в ClickHouse?

На практике используют batch `INSERT` (через native/http клиент), загрузку из файлов (`INSERT ... FORMAT`, `clickhouse-client`), ingestion через Kafka engine + materialized views, а также коннекторы ETL-инструментов/Spark. Ключевой фактор — размер батча и стабильность потока.

Типичный production-паттерн: писать данные сначала в сырую локальную таблицу, затем агрегировать/дедуплицировать в витринную таблицу, чтобы уменьшить влияние late data и мутаций на запросы пользователей.

- [ ] Что важно знать про ReplacingMergeTree?

Он удаляет дубли не мгновенно, а во время merge. Поэтому «сразу после insert» можно увидеть старые и новые версии одновременно.

Для корректной аналитики либо строят запросную логику дедупа (`argMax` и т.д.), либо используют `FINAL` точечно (понимая его цену).

- [ ] Как в ClickHouse работают UPDATE/DELETE?

Это мутации, то есть асинхронная перепись частей, а не легкий row-level update как в OLTP. На больших объемах это дорого.

Поэтому в аналитических системах часто проектируют pipeline как insert-only с периодической перезаписью партиций.

- [ ] Что такое Distributed-таблица?

Это логический слой маршрутизации, который отправляет запросы на локальные таблицы в шардах кластера. Данные обычно лежат в локальных MergeTree-таблицах.

Правильное шардирование и понимание сетевого поведения join критичны для предсказуемой производительности.

- [ ] Почему обычный JOIN по Distributed-таблицам может дать неполный результат?

Локальный join выполняется на каждом шарде только по локальным кускам; если нужная строка правой таблицы лежит на другом шарде, локальная пара не строится. Для полного соединения иногда нужен `GLOBAL JOIN` (дорого: вся правая таблица на все шарды) или заранее колокировать данные по ключу шардирования.

Поэтому в ClickHouse проектируют шард-ключ и витрины так, чтобы критичные join были локальными.

- [ ] Зачем в ClickHouse Dictionary и когда он лучше обычного JOIN?

Dictionary — заранее загруженный в память (или кешируемый) справочник по ключу для быстрого обогащения строк без тяжёлого shuffle между узлами. Уместен, когда правая сторона относительно небольшая и читается много раз.

Если справочник не влезает в RAM или часто меняется, используют нормальную таблицу + пайплайн обновления.

- [ ] Что такое TTL в ClickHouse?

Правила на уровне таблицы/колонок: удалить или переместить данные старше порога (или по выражению). Работает фоном, помогает retention и выгрузке холодных партиций на S3.

Нужно понимать асинхронность и взаимодействие с merge — это не OLTP-delete по одной строке.

- [ ] Зачем ASOF JOIN и SEMI JOIN в ClickHouse?

`ASOF JOIN` сопоставляет строки по ближайшему предыдущему времени (курс на момент сделки, сенсор к моменту события). `SEMI`/`LEFT SEMI` оставляет строки левой таблицы, если есть хотя бы одно совпадение в правой — удобная замена `EXISTS`/`IN` без раздувания дубликатов.

`ANTI` — анти-полуприсоединение для отсутствия пары.

- [ ] Видны ли строки сразу после `ALTER DELETE` в ClickHouse в SELECT?

По умолчанию мутации асинхронны: до завершения переписывания частей старые строки могут ещё читаться. Настройки `mutations_sync` и версия движка влияют на ожидание; для строгой семантики часто полагаются на перезапись партиции или на запросную дедупликацию.

В проде важно не строить онлайн-отчёты, предполагающие мгновенный hard delete.

- [ ] Как смоделировать удаление в ReplacingMergeTree?

Вставляют новую версию строки с тем же ключом сортировки и колонкой версии (`ver`) или флагом `is_deleted=1`, а в витрине берут «последнюю живую» через `argMax`/группировку или точечно `FINAL`, понимая стоимость.

Физическое исчезновение строки происходит только на merge.

- [ ] Всегда ли выгодно шардирование в ClickHouse?

На небольших объёмах сеть scatter/gather и сложность join перевешивают выигрыш. Шардирование оправдано, когда данных много, нагрузка чтения распределена и запросы укладываются в локальные паттерны.

Иначе лучше одна мощная реплика или вертикальное масштабирование до порога.

- [ ] Можно ли масштабировать чтение за счёт реплик ClickHouse?

Да: настройки балансировки на стороне клиента/прокси (`load_balancing`) распределяют SELECT по репликам одного шарда, снимая CPU с одной ноды. Запись и merge всё равно координируются через репликационный очередь/Keeper.

Это не замена шардирования для данных, которые не помещаются на один узел.

- [ ] Есть ли «мастер» записи в ClickHouse-кластере?

С ReplicatedMergeTree классического single-writer master нет: любая реплика может принимать insert при корректной конфигурации, синхронизация идёт через Keeper/ZooKeeper. «Кто мастер» — распределённая роль лидера только для отдельных фоновых операций, не как у некоторых legacy СУБД.

Формулировка на интервью: multi-master на уровне реплик с координацией метаданных.

- [ ] Что такое гранула (granule) в ClickHouse?

Гранула — блок строк, по которому хранится sparse-индекс. Движок использует гранулы для приблизительного отсечения данных при чтении.

Это не B-tree индекс как в OLTP, поэтому модель оптимизации в ClickHouse другая: важны порядок данных и skipping, а не «индекс на каждую колонку». Параметр `index_granularity` задаёт размер гранулы: меньше — точнее отсечение, но больше метаданных marks и нагрузка на merge; дефолт порядка 8192 обычно разумен без отдельного профилирования.

- [ ] Какие системные таблицы полезны для диагностики в ClickHouse?

Часто смотрят `system.query_log`, `system.parts`, `system.merges`, `system.mutations`, `system.replicas`. По ним видно медленные запросы, состояние слияний, мутаций и репликации.

На интервью это хороший маркер production-опыта: ты умеешь не только писать SQL, но и расследовать инциденты.

- [ ] Что такое Sparse Index в ClickHouse?

В ClickHouse primary index разреженный: он хранит метки по гранулам, а не указатель на каждую строку как в B-tree. За счет этого индекс компактный и быстрый на больших объемах, но работает лучше при правильном `ORDER BY`.

Важно понимать, что это не OLTP-индекс точечного поиска, а механизм грубого отсечения диапазонов.

- [ ] Какие движки MergeTree важно различать?

`MergeTree` — базовый движок, `ReplacingMergeTree` — дедупликация версий при merge, `SummingMergeTree` — предагрегация суммируемых метрик, `AggregatingMergeTree` — хранение aggregate states.

Выбор движка задает стратегию хранения и перерасчета, поэтому должен исходить из паттернов запросов и обновлений.

- [ ] Чем Materialized View отличается от обычного VIEW в ClickHouse?

Обычный VIEW — это просто сохраненный SQL, который исполняется при чтении. Materialized View в ClickHouse срабатывает при INSERT в источник и записывает результаты в целевую таблицу.

Поэтому MV — инструмент precompute/ingestion, а не только логический синоним запроса.

- [ ] Чем materialized view в ClickHouse принципиально отличается от materialized view в PostgreSQL?

В PostgreSQL materialized view — это сохранённый результат запроса, который обновляют явной командой `REFRESH` (или расширениями). В ClickHouse типичная MV — триггер на поток вставок: при `INSERT` в исходную таблицу движок исполняет `SELECT` и пишет в другую таблицу (часто с другим `ENGINE`), то есть это скорее ingestion-пайплайн, а не кэш одного запроса.

На интервью важно не переносить ожидания «как в Postgres» без перечитывания семантики merge и задержек.

- [ ] Почему в `ReplacingMergeTree` / `CollapsingMergeTree` дубликаты могут быть видны сразу после вставки?

Дедупликация и «схлопывание» пар выполняются при фоновом merge частей, а не синхронно на каждый `INSERT`. Пока части не смержились, `SELECT` видит несколько версий строки.

Для строгой семантики используют запросную логику (`argMax`, осторожно `FINAL`) или проектируют поток без ожидания мгновенного merge.

- [ ] Как по `EXPLAIN` понять, что распределённый запрос ClickHouse тянет данные с шардов на инициатор?

В плане появляются шаги вроде `Remote` / чтение с удалённых нод и этапы gather/обмена, когда запрос идёт к `Distributed`-таблице или кластерной функции. Если фильтр попадает в ключ шардирования и запрос локализуется одним шардом, сетевого чтения со всех нод меньше.

Это аналог понимания motion в MPP: дорого не столько «join», сколько пересылка данных.

- [ ] Как приблизить ClickHouse к near real-time без тяжёлых batch-ETL?

Типичная связка: `Kafka` engine + `MATERIALIZED VIEW` в целевую `MergeTree`, крупные батчи вставки, при необходимости `Buffer` перед основной таблицей и мониторинг лагов consumer/мержей. Latency задаётся размером батча, скоростью merge и политикой дедупа, а не одной галочкой.

Для строгих SLA добавляют отдельную витрину «почти свежую» и тяжёлые агрегаты — в отложенном слое.

- [ ] Почему мелкие INSERT в ClickHouse — зло?

Каждая вставка создает part, и при слишком частых мелких батчах система тратит много ресурсов на merge-обслуживание вместо запросов. Это увеличивает latency и нестабильность кластера.

Лекарство — батчировать вставки, контролировать throughput и проектировать ingestion-поток под "крупнее и реже".

- [ ] Как читать данные из S3/MinIO напрямую в ClickHouse?

Обычно используют table function `s3()`/`s3Cluster()` с указанием пути, формата и схемы, либо внешние таблицы/пайплайн через ingestion layer. Это удобно для ad-hoc чтений и bootstrap-загрузок.

Для прод-стабильности лучше фиксировать схему и контролировать партиционирование/формат заранее.

## 12) dbt, lakehouse, Trino, Kubernetes, метаданные

- [ ] Что такое dbt и где его место в пайплайне?

dbt (data build tool) — это инструмент трансформаций «SQL как код»: вы описываете модели (что построить в БД), sources (откуда читать сырьё), tests (ограничения качества), docs и lineage — всё в git с code review. Типичное место в стеке: после ingestion (Raw/Silver уже в warehouse/lakehouse), до финального потребления BI/ML (Gold‑витрины). `ref()` собирает DAG зависимостей между моделями.

dbt не заменяет Airflow/Dagster (это всё равно оркестратор ставит задачу `dbt run`), не заменяет Spark ingestion из Kafka и не является единственным ответом на streaming.

- [ ] Что такое Iceberg в двух фразах?

Apache Iceberg — открытый табличный формат поверх объектного хранилища (совместим с Spark, Trino, Flink…). Он хранит данные файлами (Parquet и др.), но добавляет богатые метаданные слоя таблицы: snapshots (версии), manifest-файлы, partition evolution, schema evolution, time travel (прочитать таблицу «как вчера»), конкурентные коммиты без «сломали каталог случайным `rename`».

Для DE это шаг от «просто папки в S3» к lakehouse: появляются upsert/delete на уровне формата, управление мелкими файлами (compaction), более безопасные сценарии с несколькими писателями при дисциплине эксплуатации каталога.

- [ ] Что такое Trino и когда он нужен?

Trino (ранее PrestoSQL) — распределённый SQL‑движок (MPP‑исполнитель запросов) для federated querying: можно JOIN‑ить Hive/Iceberg в lake, таблицы PostgreSQL, Kafka‑коннекторы и другие источники в одном SQL.

Польза для adhoc/исследования и единого интерфейса. Риски: не всегда есть predicate pushdown в слабый источник, сеть утянет дорогущие строки на кластер, консистентность между системами не «магическая». Нужен SLA/понимание задержки реплик.

- [ ] Как Kubernetes связан с Data Engineering?

Kubernetes (K8s) — платформа оркестрации контейнеров (pods/deployments/…). В DE на нём часто крутят Airflow (Celery/KubernetesExecutor), Spark на K8s, Flink/Kafka Connect, dbt в CI job, Argo/Tekton для пайплайнов. Важны requests/limits CPU/RAM, HPA/cluster autoscaler, node pools под тяжёлые ворклоады, ретраи, graceful shutdown, централизованные logs/metrics.

Ответ интервью: «K8s даёт воспроизводимую упаковку образа и эластичный compute для batch/stream задач».

- [ ] Чем Docker-контейнер отличается от виртуальной машины с точки зрения DE?

Контейнер делит ядро Linux (или Windows контейнеры) с хостом; изоляция через namespaces и cgroups. Образ тянет только слои приложения — секунды на старт. Виртуальная машина несёт полную ОС, hypervisor, сильнее изоляция и периметр безопасности, но тяжелее по ресурсу и cold start.

Практически: Dockerfile описывает зависимости задачи один раз; образ гоняется локально как в проде‑кластере (в идеале).

- [ ] Зачем нужен каталог метаданных (OpenMetadata и аналоги)?

Data catalog (например OpenMetadata, DataHub, Collibra, Purview) — это единая точка про datasets: описание смыслов, владелец домена, теги PII, линия происхождения (lineage из Airflow/dbt/ELT‑инструментов), поиск таблицы «где живёт churn», SLA качества/связь с тикетами.

Для бизнеса это снижает время onboarding аналитиков; для DE и Governance — способ связать изменение схемы с списком потребителей (impact analysis).

- [ ] Что такое Incremental Models в dbt и как они работают?

Incremental model обновляет только новые/измененные данные вместо полного пересчета таблицы. Под капотом стратегия зависит от адаптера: `merge`, `delete+insert`, insert overwrite partition и т.д.

Ключевое — правильно определить уникальный ключ и условие инкремента, иначе легко получить дубли или пропуски.

- [ ] Как тестировать данные в dbt?

Базовые generic tests — `unique`, `not_null`, `accepted_values`, `relationships`; дополнительно часто пишут custom tests под бизнес-правила. Тесты выполняются в CI/пайплайне и дают быстрый сигнал о деградации качества.

Важна не только "наличие тестов", но и их приоритизация: какие блокируют деплой, какие лишь предупреждают.

- [ ] Что такое Lineage в dbt?

Lineage — граф зависимостей моделей, который строится через `ref()`/`source()`. Он показывает путь данных от источников до витрин и помогает локализовать влияние изменений.

Для собеседования это хороший мост между инженерией и governance: проще делать impact analysis и объяснять происхождение метрик.

- [ ] Что такое Data Observability?

Data Observability — практики и инструменты для непрерывного контроля состояния данных и пайплайнов: freshness, volume, schema drift, quality anomalies, SLA. Это развитие идеи мониторинга "не только джобы, но и сами данные".

Примеры подходов: dbt tests, Great Expectations, платформы вроде Monte Carlo, кастомные checks в оркестраторе.

- [ ] Какие измерения Data Quality обычно выделяют?

Часто используют completeness, uniqueness, validity, consistency, timeliness/freshness, accuracy. Эти измерения помогают формализовать качество и привязать его к алертам и SLA.

Важно выбирать метрики качества по критичности домена, а не пытаться контролировать всё одинаково строго.

- [ ] Как отслеживать Freshness и Volume Drift?

Freshness обычно контролируют по max(event_time/load_time) против SLA; volume drift — сравнение объема с baseline (день-к-дню, неделя-к-неделе, статистические пороги). Оба сигнала полезны как ранние индикаторы поломки ingestion.

На практике качество лучше ловить проактивно до публикации витрины, а не после жалобы пользователей.

- [ ] Что такое Schema Evolution и как безопасно переживать добавление новой колонки?

Schema evolution — управляемое изменение структуры данных со временем. При добавлении поля нужны совместимые схемы, fallback/default значения и проверка downstream, чтобы не сломать существующие консьюмеры.

Хорошая практика — контракт, тесты и staged rollout: сначала поддержка новой схемы, потом использование нового поля в бизнес-логике.

- [ ] Что такое DLQ (Dead Letter Queue) в data pipelines?

DLQ — отдельный поток/хранилище для событий, которые не удалось обработать корректно (битая схема, валидация, трансформация). Это позволяет не блокировать основной поток и разбирать ошибки отдельно.

Важно хранить причину ошибки и контекст, чтобы можно было сделать replay после исправления.

- [ ] [Стриминг / lakehouse] Опоздавшее событие и watermark

Вопрос: «Событие по event time должно попасть в окно агрегации, но оно опоздало сильнее, чем watermark позволяет учесть. Что делать?»

Зачем спрашивают
Проверяют понимание границ оконной семантики, state и компромиссов с бизнесом по поздним данным.

🔍 Суть проблемы
Watermark задаёт предел опоздания: состояние старого окна может быть очищено, когда водяная метка ушла вперёд.

Сверхопозднее событие не попадает в ту же инкрементальную агрегацию без отдельных механизмов.

Риск «молча потерять» событие для скользящего дашборда, если SLA поздней доставки не согласован.

📖 Пояснения к определениям
— event time — время события в домене, а не время прихода записи на обработчик.
— watermark — оценка «какое event time мы уже считаем завершённым» с допуском на опоздания.
— allowed lateness — дополнительное время, в течение которого окно ещё может принять опоздавшие записи после первого триггера.

💬 Как отвечать на собеседовании (структура ответа)

«Сначала согласую с бизнесом SLA поздней доставки событий. Дальше варианты: расширить allowed lateness и watermark — дороже по state и задержке выхода результатов; ввести DLQ или таблицу late_events для ручной дорожки; пересчитать витрину из архива в lake идемпотентный MERGE. Часто делают два уровня: быстрый приблизительный поток и ночной точный пересчёт по сырью.»

- [ ] [Стриминг / lakehouse] Дубликаты при at-least-once

Вопрос: «Откуда берутся дубликаты при at-least-once и как с этим жить?»

Зачем спрашивают
Проверяют гарантии доставки и проектирование идемпотентного приёмника.

🔍 Суть проблемы
At-least-once: сообщение может быть обработано повторно (краш до commit, retry, rebalance группы).

Приёмник должен уметь принять повтор без повторного эффекта в бизнес-метриках.

Exactly-once «сквозь» произвольный sink возможен только с поддержкой инфраструктуры и обычно дороже в эксплуатации.

📖 Пояснения к определениям
— at-least-once — каждое сообщение будет доставлено минимум один раз; возможны повторы.
— MERGE / UPSERT — запись по ключу без дубля бизнес-строки при повторной доставке.

💬 Как отвечать на собеседовании (структура ответа)

«Дубликаты — норма при at-least-once; я проектирую приёмник идемпотентным: естественный или технический ключ, MERGE/UPSERT в таблицу или в Iceberg/Delta, дедуп в промежуточном слое. Короткая формула: повтор возможен → запись обязана быть идемпотентной. Exactly-once обсуждаю только когда есть EOS во Flink/Kafka Streams/транзакционный sink.»

- [ ] [Стриминг / lakehouse] Offset commit до и после обработки

Вопрос: «Чем опасно делать offset commit до обработки сообщения и после?»

Зачем спрашивают
Проверяют понимание at-most-once vs at-least-once без маркетинга.

🔍 Суть проблемы
Commit до обработки: при падении можно потерять сообщение до повторной обработки.

Commit после обработки: возможны повторы при падении после обработки, но до записи оффсета.

📖 Пояснения к определениям
— offset commit — фиксация прогресса чтения в consumer group после успеха по выбранной политике.

💬 Как отвечать на собеседовании (структура ответа)

«Коммит до обработки склоняется к at-most-once — риск silent loss при сбое. Коммит после — at-least-once с возможными дубликатами на приёмнике. Выбор осознанный: либо терпим дубли и делаем идемпотентный sink, либо недопускаем повтор за цену риска потери. Объясняю это связкой с наблюдаемостью по лагам и ошибками.»

- [ ] [Стриминг / lakehouse] Enrichment в стриме

Вопрос: «Как обогащать (enrich) стриминговые события справочниками и контекстом?»

Зачем спрашивают
Проверяют паттерны join поток–поток, lookup в БД, broadcast и границы state.

🔍 Суть проблемы
Stream–stream join требует водяных знаков и ограниченных интервалов — иначе state растёт бесконечно.

Lookup во внешние системы упирается в latency, рейт-лимы и доступность.

Нужно разделить малый меняющийся справочник и большую медленную систему записи.

📖 Пояснения к определениям
— enrichment / enrich — дополнить событие полями из справочника или второго источника до записи downstream.
— changelog DIM — поток изменений измерения вместо периодического full snapshot.

💬 Как отвечать на собеседовании (структура ответа)

«Комбинирую подходы: stream-stream join по ключу и ограниченному интервалу с watermark; Async I/O + TTL-кэш или Redis для внешних API; broadcast state для малого DIM или changelog DIM через отдельный CDC-топик. В ответе называю источник правды для справочника, частоту обновления, размер и порядок относительно фактов.»

- [ ] [Стриминг / lakehouse] Параллелизм, обработка на лету и запись в хранилища

Вопрос: «Зачем параллелить поток, обрабатывать на лету и одновременно писать в хранилище?»

Зачем спрашивают
Проверяют связь параллелизма партиций Kafka, операционной пользы near real-time и паттернов sink.

🔍 Суть проблемы
Throughput растёт за счёт независимых ключей/partition без нарушения порядка внутри партиции.

На лету — правила, агрегации окна, алертинг минутной картины.

Sink часто буферизуют во избежание small files и блокировки горячего пути синхронным удалённым API.

📖 Пояснения к определениям
— throughput — пропускная способность при обработке потока при заданной задержке.

💬 Как отвечать на собеседовании (структура ответа)

«Разные ключи обрабатываются разными задачами параллельно при сохранении порядка внутри партиции. Обработка на лету нужна для бизнес-сигналов с небольшой задержкой. Запись в хранилище делаю батчами: микробатчи в sink, буферные таблицы в OLAP где уместно, checkpoint перед коммитом в lake. Часто два канала: горячий для низкой задержки и холодный для полного аудита в lake без блокировки конвейера.»

- [ ] [Стриминг / lakehouse] Stream-stream JOIN без проектирования

Вопрос: «Stream-stream JOIN: что может пойти не так без проектирования?»

Зачем спрашивают
Проверяют упорядочивание между источниками и рост состояния.

🔍 Суть проблемы
События приходят в разном порядке и с разной задержкой.

Без ограниченного временного интервала join не сходится или раздувает state.

📖 Пояснения к определениям
— interval join / temporal join в стриме — соединение потоков с ограничением «на сколько времени назад искать вторую запись».

💬 Как отвечать на собеседовании (структура ответа)

«Ставлю interval join или эквивалент с ограничением времени совпадения, выравниваю watermark на обоих источниках и явные границы хранения state. По сути это join по ключу и временному окну совместности, не «наивный JOIN как в SQL batch». Объясняю риски: нет матчей или неконтролируемая память.»

- [ ] [Стриминг / lakehouse] Out-of-order и watermark

Вопрос: «Что такое out-of-order события и как они связаны с watermark?»

Зачем спрашивают
Проверяют модель времени и политики отброса поздних записей.

🔍 Суть проблемы
Out-of-order: более старое по event time сообщение приезжает позже более нового уже обработанного.

Watermark задаёт порог замкнутости окна при допущенном опоздании.

Дальнейшая судьба опоздавших строк — политика: drop, allowed lateness, DLQ.

💬 Как отвечать на собеседовании (структура ответа)

«Отвечу так: watermark — допуск между «уже считаем результат закрытым» и реальной задержкой сети. Out-of-order внутри допуска ещё учитываются; если нет — идём в политики late handling и согласование с SLA. Связываю с наблюдением по доле поздних событий для тюнинга.»

- [ ] [Стриминг / lakehouse] Small files в lake из стрима

Вопрос: «Как избежать small files при непрерывной записи в lake из стрима?»

Зачем спрашивают
Проверяют баланс латентности стрима и размера commit в объектном хранилище.

🔍 Суть проблемы
Каждый микрокоммит даёт файл; без слияния растёт метадата и тормозит чтение.

Компактика и разумное партиционирование — часть эксплуатации, не только настройки джоб.

📖 Пояснения к определениям
— small files problem — много крошечных файлов в озере ухудшает планирование и metadata overhead.
— compaction — слияние мелких файлов в крупнее для производительности чтения.

💬 Как отвечать на собеседовании (структура ответа)

«Увеличиваю размер записываемой пачки за счёт micro-batch триггеров или буфера; планирую scheduled compaction или optimize в Iceberg/Delta по регламенту; выбираю partitioning без взрыва числа активных ключей записи параллельно. Явно называю компромисс: ниже latency стрима — больше мелких файлов без компенсирующей компактики.»

- [ ] [Стриминг / lakehouse] Зачем replay и backfill при наличии DLQ

Вопрос: «DLQ уже есть — зачем ещё replay или отдельный backfill job?»

Зачем спрашивают
Проверяют различение точечной ошибки и массового пересчёта.

🔍 Суть проблемы
DLQ ловит битые или неугодные по правилам записи точечно.

При смене логики, багах или месячном переобработчике нужен контролируемый reprocess истории.

📖 Пояснения к определениям
— DLQ (Dead Letter Queue) — очередь/таблица для сообщений, которые нужно разобрать вне основного потока.
— replay / backfill — повторная обработка диапазона данных по новым правилам или после сбоя.

💬 Как отвечать на собеседовании (структура ответа)

«DLQ закрывает известную плохую запись после разбора. Replay и backfill — когда нужно переиграть период с новой версией трансформации, закрыть дыру между batch и потоком или подтянуть архив переотправкой. Отделяю SLA и идемпотентность ключей записи под повторную запись того же интервала.»

- [ ] [Стриминг / lakehouse] Hot path и cold path одной фразой

Вопрос: «Одной фразой: hot path против cold path в потоково-пакетной архитектуре.»

Зачем спрашивают
Проверяют язык эксплуатации конвейера с двумя скоростями корректности.

🔍 Суть проблемы
Два канала конфликтуют если нет общего контра ключа строки между ними.

💬 Как отвечать на собеседовании (структура ответа)

«Hot path даёт малую latency и упрощаемую семантику с watermark. Cold path доводит полную точность по архиву данных с ночной или офлайн-идемпотентностью. Финальный слой связывает их через ключ записи и согласованные метрики.»

- [ ] [Стриминг / lakehouse] Schema drift в потоке

Вопрос: «Появился schema drift в потоке: что ответить?»

Зачем спрашивают
Проверяют управление совместимостью схем и эволюцией без даунтайма всех потребителей.

🔍 Суть проблемы
Новые поля и типы ломают десериализацию, если не обеспечить совместимость.

Критические ломания требуют отдельного топика или этапов деплоя.

📖 Пояснения к определениям
— schema drift — постепенное расхождение схемы продюсера и ожиданий консьюмеров.

💬 Как отвечать на собеседовании (структура ответа)

«Опираюсь на Schema Registry с политикой backward/forward compatibility, дефолты в коде декодера и тестовый контур consumer contract. Серьёзные изменения версионирую или вывожу в новый топик с миграцией клиентов. Использую feature flags чтобы поэтапно включить использование нового поля.»

- [ ] [Стриминг / lakehouse] Backpressure в стриме

Вопрос: «Backpressure: зачем нужен и где возникает в стриминговой обработке?»

Зачем спрашивают
Проверяют понимание разницы между скоростью продюсера и способности sink и как не упереться в OOM.

🔍 Суть проблемы
Без торможения производителя очередь в памяти растёт до срыва процесса.

Причины: тяжёлый join, медленное внешнее API, bottleneck на записи.

📖 Пояснения к определениям
— backpressure — сигнал «замедлись» upstream когда downstream не успевает, чтобы не распухала буферная память.

💬 Как отвечать на собеседовании (структура ответа)

«Backpressure нужен когда upstream быстрее чем успевают обрабатывать или писать downstream. Инструменты: уменьшить fetch/partition parallelism осторожно, масштабировать consumer, оптимизировать join, ставить ограниченные очереди на async записи или вынос медленной части на отдельный холодный путь. Связываю с наблюдением backlog и временем обработки на стадиях движка.»

- [ ] [Диасофт] Новые источники, пайплайны, DQ и наблюдаемость

Вопрос: «Новые источники появляются часто — как организовать пайплайны, DQ и наблюдаемость без хаоса?»

Зачем спрашивают
Это классический вопрос на масштабируемость архитектуры и инженерную зрелость. Интервьюер хочет понять: сможете ли вы построить систему, которая не развалится, когда источников станет не 5, а 50.

🔍 Суть проблемы
В реальных проектах источники появляются постоянно:
- новые партнёры, сервисы, мобильные приложения;
- изменения в существующих системах (новые колонки, форматы);
- экспериментальные фичи, которые могут вырасти в продукт.

Риск, если каждый источник — «уникальный» ручной DAG:
- зоопарк пайплайнов, которые тяжело сопровождать;
- инциденты из-за хрупких зависимостей;
- падает доверие к данным;
- команда занята поддержкой вместо развития.

Цель: стандартизированный контур, где подключение источника — в основном конфигурация и контракт качества, а не бесконечные «уникальные» скрипты.

📖 Пояснения к определениям
— DAG Factory — один код/шаблон генерирует десятки похожих DAG из конфигов источников.
— DQ (Data Quality) — набор проверок и политик: что считается «годными» данными на границе слоёв.
— RAW / CORE / MART — сырьё → нормализованное/мастер-ядро → витрины под продукты и отчёты.
— Iceberg / Delta — форматы/табличные протоколы lakehouse поверх объектного хранилища с ACID-семантикой на уровне таблицы.

💬 Как отвечать на собеседовании (структура ответа)

«Чтобы масштабировать ingestion без хаоса, я опираюсь на пять принципов:

1) Конфигурационная загрузка: новые источники описываются в конфиге (YAML/таблица метаданных), а генератор пайплайнов (DAG Factory) воспроизводит одинаковую структуру задач. Это сокращает time-to-onboarding с дней до часов.

2) Слои с DQ-контрактами: RAW → CORE → MART, на каждом переходе — обязательные проверки: схема, ключи и уникальность там, где нужно, своевременность (freshness), бизнес-правила там, где возможно. Провал — алерт и останов продвижения «грязи» ниже по слою.

3) Централизованная наблюдаемость: единые дашборды (latency job, SLA, ошибки задач), алерты в мессенджер при breach; владельцы источников и пайплайна понятно назначены.

4) Lakehouse-паттерны при росте: Iceberg/Delta дают транзакционность слоя хранения, time travel, безопасную конкуренцию чтение/запись — но внедряю осознанно, не как магию.

5) Изоляция исследований: песочница и отдельные ветки для ad-hoc, чтобы эксперименты не ломали прод.

Итог: новый источник — рутинная операция с контролем качества, а не героический подвиг.»

- [ ] [Альфа-инновация] Шардирование и скорость запросов

Вопрос: «Шардирование всегда ускоряет запросы?»

Зачем спрашивают
Проверяют, понимаете ли вы стоимость распределённых систем: когда горизонтальное масштабирование помогает, а когда добавляет сетевой scatter–gather и ломает план запроса.

🔍 Суть проблемы
Локальный запрос по шарду может стать быстрее за счёт параллелизма и данных на дисках ближе к CPU.

Кросс-шард запросы тянут пересылку данных и координацию — иногда медленнее «одной хорошей машины».

Неправильный shard key даёт перекос и «длинный хвост» задач.

📖 Пояснения к определениям
— Шардирование — горизонтальный сплит данных по узлам кластера; запрос может затрагивать один или много шардов.
— Scatter–gather — план: разослать работу по узлам и собрать частичные результаты на координаторе.

💬 Как отвечать на собеседовании (структура ответа)

«Сначала смотрю профиль запросов: они фильтруются по шард-ключу или нет. Ускорение вероятно, если запросы локальны. Если типичный отчёт собирает полкластера — шардирование само по себе не панацея; важнее модель данных, колокация join и стоимость сети.»

Concurrent run двух задач на одну партицию даёт race.

💬 Как отвечать на собеседовании (структура ответа)

«Смотрю класс ошибки и логи задачи/pool limits. Проверяю idempotency: можно ли просто переиграть задачу или нужен backfill интервала. При блокировках в БД координирую очистку. Не запускаю второй конфликтующий пайплайн без правил промоута.»

- [ ] [Сбер IBS] Ожидаемые размеры JOIN

Вопрос: «Зачем на собеседовании спрашивают про количество строк после разных JOIN?»

Зачем спрашивают
Проверяют понимание fan-out, NULL после OUTER и sanity-check моделей.

🔍 Суть проблемы
Случайный INNER с дублями ключей многократно раздувает факты.

📖 Пояснения к определениям
— Fan-out — рост числа строк при join из-за отношения много-ко-многим или дублей.

💬 Как отвечать на собеседовании (структура ответа)

«Это санити-контроль: перед trust витрине я считаю ожидаемые размерности. LEFT сохраняет левые строки, INNER режет кардинальность к пересечению; CROSS — декартово. Неожиданный размер ⇒ проверить unique key, фильтры и тест на дубликаты join keys.»
